# v0.2.4 迭代设计方案

## 概述

10 个功能点，覆盖体验精打磨、效率工具、协作分享、智能化增强、性能与稳定性五个方向。

## 通用约定

所有新功能必须处理以下边界状态：
- **空状态**：使用 `PhaseEmptyState` 或文字提示
- **加载态**：使用 Skeleton 组件或按钮 disabled + loading 文案
- **错误态**：使用 Toast error 通知 + console.error 日志（E2 完成后接入统一错误处理）

新功能嵌入阶段页面时，须遵循现有 `PhaseShell` 布局结构（header → context → content → footer），不在各页面中另建布局。

## 1. A4 — Settings 页拆分

**现状**：Settings.tsx 1232 行，所有配置挤在一个滚动页面，维护困难。

**方案**：拆成 3 个 Tab 子页面组件。

| Tab | 内容 |
|-----|------|
| **API 配置** | API Key 检测与配置、Base URL、Model 选择、Backend 切换（API/CLI）、连通性测试 |
| **项目管理** | 项目输出目录选择、默认项目配置、上下文记忆开关（D2） |
| **关于** | 版本号、更新检查与下载、环境依赖检测、错误日志查看（E2）、数据迁移（条件显示） |

**数据迁移处理**：迁移功能不独立成 Tab（迁移完成后变空白页无意义）。改为「关于」Tab 下的可折叠区块，扫描到 legacy 数据时才展示，迁移完成后自动隐藏。

**实现细节**：

- Tab 栏使用水平标签导航，样式复用 Review 页的角色 Tab 模式
- 使用 `useSearchParams` 管理 Tab 状态（如 `/settings?tab=project`），支持外部跳转（如从 onboarding 引导到 API 配置）和刷新保持
- 每个 Tab 内容拆为独立组件文件：`SettingsApi.tsx`、`SettingsProject.tsx`、`SettingsAbout.tsx`
- `Settings.tsx` 退化为 Tab 壳 + 条件渲染，约 60 行

**改动文件**：
- 拆分：`app/src/pages/Settings.tsx` → 1 个壳 + 3 个子组件
- 新建：`app/src/pages/settings/SettingsApi.tsx`、`SettingsProject.tsx`、`SettingsAbout.tsx`

## 2. B3 — Dashboard 批量操作

**现状**：Dashboard 只能对单个项目右键操作（重命名、删除），无法批量处理。

**方案**：在 Dashboard 增加多选模式 + 底部批量操作栏。

**交互流程**：

```
正常模式
  → 标题栏右侧「选择」按钮 / Cmd+点击卡片 → 进入多选模式
  → 卡片左上角出现 checkbox，底部浮现批量操作栏
  → 操作按钮：归档 | 导出 | 删除
  → 点击操作 → 确认对话框 → 执行 → 退出多选模式
```

**多选模式触发**（桌面端交互，不用长按）：
- 页面标题栏右侧「选择」按钮：点击进入/退出多选模式
- Cmd/Ctrl + 点击：切换单张卡片选中状态（自动进入多选模式）
- Shift + 点击：范围选择（从上次选中到当前）
- Space：切换当前聚焦卡片的选中状态
- Esc 退出多选模式

**批量操作栏**：
- 固定在页面**底部**（类似 macOS Finder 底部状态栏），毛玻璃背景，不遮挡搜索/筛选栏
- 左侧：全选 checkbox +「已选 {N} 个项目」（`aria-live="polite"` 播报变化）
- 右侧：归档按钮、导出按钮（点击后按钮变 disabled +「导出中...」态）、删除按钮（destructive + 二次确认）

**可访问性**：
- ProjectCard 多选模式下增加 `role="option"` + `aria-selected={isSelected}`
- 卡片列表容器增加 `role="listbox"` + `aria-multiselectable="true"`

**批量导出**：
- 选中项目的 output 目录打包为 `.zip`
- 后端新增 `export_projects_zip` 命令，接收项目 ID 列表，返回 zip 文件路径
- 对每个 output_dir 做 canonicalize + home 目录前缀校验（路径遍历防护）
- 使用 `tokio::task::spawn_blocking` 包裹 zip 操作避免阻塞主线程
- 前端调用后用 `shell:open` 在文件管理器中显示

**批量删除/归档**：
- 后端新增 `batch_delete_projects(ids: Vec<String>)` 命令，一次 Mutex lock 内用 SQLite 事务批量执行 DB 操作，事务提交后逐个删除文件系统目录（best-effort，失败不阻塞）
- 后端新增 `batch_set_project_status(ids: Vec<String>, status: String)` 命令，同样事务保护
- 返回 `{ succeeded: Vec<String>, failed: Vec<(String, String)> }`，前端据此展示结果

**前端状态管理**：
- 多选逻辑抽取为 `useBatchSelect(projects)` 自定义 Hook，内聚 selectedIds、Shift 锚点、全选/取消等
- `ProjectCard` 用 `React.memo` + 稳定的 `onToggleSelect` 回调隔离重渲染
- 统一使用 `tauri-api.ts` 中的 `ProjectSummary` 类型（消除 `DashboardProject` 重复定义）

**新增 Rust 依赖**：`zip = "2"`（Cargo.toml）

**改动文件**：
- 修改：`app/src/pages/Dashboard.tsx`（多选模式 + useBatchSelect）
- 修改：`app/src/components/ProjectCard.tsx`（选中态 + checkbox + React.memo）
- 新建：`app/src/components/BatchActionBar.tsx`
- 新建：`app/src/hooks/use-batch-select.ts`
- 后端新增：`export_projects_zip`、`batch_delete_projects`、`batch_set_project_status` 命令
- 修改：`app/src-tauri/Cargo.toml`（添加 `zip` 依赖）

## 3. C1 — PRD 一键分享

**现状**：PRD 只能在应用内查看或导出为文件（MD/PDF/DOCX），无法直接分享给他人在线查看。

**方案**：生成自包含的静态 HTML 文件，用户可通过任意方式分享（邮件、IM、内网）。

**不做**：不搭建分享服务器、不做链接托管。保持桌面应用的离线定位。

**生成内容**：
- 单文件 HTML，内联所有 CSS/JS
- Markdown 渲染方式：后端将原始 Markdown 文本嵌入 HTML 模板，内联 `marked.min.js`（~50KB）+ `mermaid.min.js` 在浏览器端渲染。**不在 Rust 端渲染 Markdown**，避免与前端 `react-markdown` 输出不一致
- 包含：TOC 目录导航（JS 自动生成）、Mermaid 图表、响应式布局
- 顶部 banner：项目名 + 生成时间 + 「由 AI PM 生成」水印
- Mermaid 配置：`securityLevel: 'strict'`（防 XSS）
- 样式：Apple HIG 风格，阅读体验优先（serif 正文字体、合理行高、舒适边距）

**交互入口**：
- PRD 页面将现有 3 个独立导出按钮（复制 MD、打印/PDF、导出 DOCX）合并为一个「导出」下拉菜单按钮，新增「生成分享页」选项
- 导出命令接受可选 `version` 参数（D3 实现后支持指定版本，默认导出最新版本）
- 生成完成后 Toast 提示 + 「在文件管理器中显示」按钮

**实现方式**：
- 后端新增 `export_prd_share_html` 命令
- 参数：`project_id: String, version: Option<u32>`
- 读取 PRD markdown 文件 → 转义嵌入 HTML 模板 → 写入 `{output_dir}/05-prd/PRD-分享页.html`
- 模板存放在 `app/src-tauri/resources/templates/share-template.html`

**改动文件**：
- 修改：`app/src/pages/project/Prd.tsx`（导出按钮合并为下拉菜单 + 新增分享选项）
- 后端新增：`export_prd_share_html` 命令
- 新建：`app/src-tauri/resources/templates/share-template.html`（内联 marked.js + mermaid.js + CSS）

## 4. D1 — 阶段智能跳过

**现状**：9 阶段强制线性流程，简单需求也要走完所有阶段，用户觉得冗长。

**方案**：需求分析完成后，后端基于确定性规则评估需求复杂度，推荐跳过不必要的阶段。用户可接受、忽略或事后恢复。

**推荐逻辑**（后端确定性规则，不依赖 AI 输出解析）：

后端新增 `suggest_skip_phases(project_id: String)` 命令，读取分析报告内容，按规则表返回推荐：

| 条件（关键词/指标匹配） | 可跳过的阶段 | 理由文案 |
|------|-------------|------|
| 分析报告中无竞品/市场/行业相关关键词 | research | 「当前需求面向内部或已知领域，无需竞品分析」 |
| 识别出的功能点 ≤ 3 个 | stories | 「功能点较少，可直接撰写 PRD」 |
| 无用户行为追踪/转化分析/数据指标相关描述 | analytics | 「当前需求以功能实现为主，未涉及数据追踪，建议跳过埋点设计」 |
| 无界面/交互/UI 相关描述 | prototype | 「当前需求以后端/API 为主，无需原型设计」 |

返回值：`Vec<{ phase: String, reason: String }>`，空数组表示建议走完整流程。

**触发时机**：analysis 阶段 `stream_done` 后，前端自动调用 `suggest_skip_phases`，有推荐则显示卡片。

**交互设计**：

- 推荐卡片位置：分析结果与 InlineChat 之间，钴蓝左边框卡片样式（与 PRD assist confirmation banner 一致）
- 卡片内容：逐条列出建议跳过的阶段及理由
- 三个按钮：「采纳建议」、「保持完整流程」、关闭按钮（×）
- 采纳后卡片折叠为一行摘要：「已跳过 N 个阶段」，可展开查看

**恢复机制**：
- 侧边栏中 `skipped` 阶段显示为灰色 + ⏭ 图标
- 用户点击 skipped 阶段 → 正常导航到对应页面 → 页面顶部提示「此阶段已标记为跳过」+ 「恢复并生成」按钮
- 点击恢复后，status 从 `skipped` 改回 `pending`，正常执行
- 用户触发生成时自动将 status 从 `skipped` 改为 `in_progress`

**数据层**：
- `project_phases` 表的 `status` 字段新增 `skipped` 值（TEXT 类型无 CHECK 约束，存储层无需迁移）
- `advance_phase` 修改：用 `BEGIN IMMEDIATE ... COMMIT` 事务包裹，while 循环查找下一个 `status != 'skipped'` 的阶段，所有后续阶段都 skipped 时返回 `None`（视为流程完成）
- `write_status_json`：`skipped` 状态输出为 `"skipped"`（非 true/false），确保 CLI `status-check.sh` 兼容
- `list_projects` 的 `completed_count`：skipped 阶段不计入完成度

**改动文件**：
- 后端新增：`suggest_skip_phases` 命令（`projects.rs`）
- 修改：后端 `projects.rs`（`advance_phase` 事务 + 跳过循环、`skip_phases` 批量更新状态、`write_status_json` 兼容）
- 修改：`app/src/pages/project/Analysis.tsx`（调用 suggest API + 渲染推荐卡片）
- 修改：`app/src/components/layout/Sidebar.tsx`（skipped 阶段样式）
- 修改：各阶段页面（skipped 态提示 + 恢复按钮，可抽为 PhaseShell 内置逻辑）

## 5. D2 — 上下文记忆（跨项目偏好学习）

**现状**：每个项目独立生成，AI 不了解用户的行业术语、写作偏好、常用产品模式。

**方案**：基于现有知识库系统，自动沉淀用户偏好，在生成时作为上下文注入。用户可见可控。

**不做**：不做复杂的 ML 模型训练。利用已有的知识库 + prompt 注入实现。

**偏好类型**：

| 偏好 | 来源 | 存储 |
|------|------|------|
| 行业术语 | 用户 PRD 中反复出现的专业词汇 | 知识库 `patterns` 分类 |
| 写作风格 | 已有的 PRD 样式系统（persona） | `templates/prd-styles/` |
| 产品偏好 | 用户在评审中采纳/拒绝的建议模式 | 知识库 `decisions` 分类 |

**自动沉淀**：
- 项目完成（retrospective 后）自动触发知识提取（现有 `knowledge sync` 能力）
- 提取时额外识别「用户偏好」类条目（行业术语、决策模式）
- `auto-extracted` 标记方式：文件名前缀 `auto-{slug}.md`，`KnowledgeEntry` 增加 `source: String` 字段（值为 `"manual"` 或 `"auto"`），`list_knowledge_internal` 中根据文件名前缀填充

**生成时注入**：
- **替换现有全量注入为推荐注入**：当前 `build_system_prompt` 中 `load_knowledge` 全量注入所有知识库条目。改为只注入与当前需求相关的推荐条目（减少 prompt 长度）
- 推荐逻辑提取为内部函数 `recommend_knowledge_internal(conn, output_dir, phase) -> Vec<KnowledgeEntry>`（不走 Tauri command，避免重复获取 Mutex 锁导致死锁）
- 注入上限：最多 5 条、总长度不超过 2000 字符
- `stream_done` 事件 payload 中返回实际注入的条目 ID 列表

**用户可见可控**：
- ContextPills 区域新增「知识库上下文」栏，列出本次注入的知识条目摘要（点击展开全文）
- 每条可单独排除（复用已有的 `excludedContext` 机制，key 格式 `knowledge:{id}`）
- 注入为空时不显示该栏
- Settings → 项目管理 Tab 新增「上下文记忆」开关（默认开启）
- 知识库页面中 `auto` 来源条目有特殊标记（自动提取徽章），用户可删除

**改动文件**：
- 修改：后端 `stream.rs`（`build_system_prompt` 改为推荐注入 + `stream_done` payload 增加注入条目）
- 新增：后端 `knowledge.rs` 中 `recommend_knowledge_internal` 内部函数
- 修改：后端 `knowledge.rs`（`list_knowledge_internal` 增加 source 字段填充）
- 修改：`app/src/pages/tools/Knowledge.tsx`（auto 来源标记 UI）
- 修改：各阶段页面的 ContextPills 区域（新增知识库上下文栏）
- 修改：Settings 项目管理 Tab（上下文记忆开关）
- 修改：`app/src/lib/stream-types.ts`（StreamDonePayload 增加 injectedKnowledge 字段）

## 6. D3 — PRD 增量迭代

**现状**：PRD 每次生成都是全量重写，无法基于 v1 + 新需求增量更新。

**方案**：支持 PRD 版本管理 + 基于前一版增量生成。

**版本管理**（文件名驱动，不加 DB 字段）：
- PRD 文件命名：`05-PRD-v1.0.md`、`05-PRD-v2.0.md`、...
- 后端新增 `list_prd_versions(project_id: String) -> Vec<String>` 命令，扫描 `05-prd/` 目录返回版本列表
- 后端新增 `get_latest_prd_version(project_id: String) -> u32` 命令
- 前端 `PRD_FILE` 常量改为函数：`prdFile(version) => \`05-prd/05-PRD-v${version}.0.md\``

**增量生成交互**（与现有 AI Assist 合并）：

现有 AI Assist 输入框增加模式切换：
- 「修改」模式（默认）：就地更新当前版本内容（现有行为）
- 「迭代」模式：保存为新版本文件（v1.0 → v2.0）

```
PRD 页面（已有 v1.0）
  → 用户在 AI Assist 输入框切换到「迭代」模式
  → 输入变更描述：「新增微信登录、去掉短信验证码」
  → AI 读取 v1.0 内容 + 变更描述 → 生成 v2.0
  → v2.0 在 PRD 页面展示，v1.0 保留在文件系统可回溯
```

现有的「重新生成」按钮行为也统一为增量迭代：保留旧版本、生成新版本。

**版本切换 UI**：
- 页面标题旁显示版本标识：「PRD 撰写 · v2.0」，点击版本号展开下拉选择其他版本
- 不独占额外空间，与合并后的导出下拉菜单配合：标题区 = `PRD 撰写 · v2.0` + `导出 ▾` + `对比`

**AI Prompt 策略**：
- 后端新增独立的 `iterate_prd` 命令（不复用 `start_stream` + `phase_config`，避免路径硬编码冲突）
- 参数：`project_id, base_version, change_description`
- 自行构建 prompt：将 base_version 全文作为 context + 变更描述 + 前置阶段输出
- v1.0 内容注入做截断保护（限制 8000 字符）
- 输出路径动态拼接：`05-prd/05-PRD-v{new_version}.0.md`

**改动文件**：
- 修改：`app/src/pages/project/Prd.tsx`（版本标识 + AI Assist 模式切换 + PRD_FILE 动态化）
- 后端新增：`list_prd_versions`、`get_latest_prd_version`、`iterate_prd` 命令
- 修改：后端 `files.rs`（多版本文件读写辅助函数）

## 7. D4 — PRD 对比 Diff

**现状**：PRD 修改后没有可视化对比，用户不知道改了什么。

**依赖**：D3（增量迭代）先实现，有多版本才有 Diff 需求。

**接口契约**：读取指定版本 PRD 内容，通过 `read_project_file(projectId, prdFile(version))` 实现，复用现有 API + D3 的版本文件命名规则。

**Diff 视图交互**：
- PRD 页面标题栏新增「对比」按钮（版本 ≥ 2 时可用）
- 点击后进入 Diff 模式：行内标注（增/删/改）
- 顶部选择对比的两个版本（默认 v(n-1) vs v(n)）

**实现方式**：
- 前端 Diff 计算：使用 `diff` npm 包，只做行级 diff（`diffLines`），不做行内字词级标注（大段修改时字词级标注难以阅读，且计算量大）
- Diff 计算用 `useMemo` 包裹，以两版本内容为依赖
- 渲染为行内标注模式：
  - 新增行：`var(--diff-add-bg)` 背景
  - 删除行：`var(--diff-del-bg)` 背景 + 删除线
- Diff 颜色通过 CSS 变量定义，适配 light/dark 主题：
  - Light：`rgba(22,163,74,0.1)` / `rgba(220,38,38,0.1)`
  - Dark：`rgba(22,163,74,0.2)` / `rgba(220,38,38,0.2)`
- Diff 视图是只读的，不可编辑

**改动文件**：
- 新增依赖：`diff` npm 包
- 新建：`app/src/components/PrdDiffViewer.tsx`（Diff 渲染组件）
- 修改：`app/src/pages/project/Prd.tsx`（对比按钮 + Diff 模式切换）
- 修改：`app/src/index.css`（Diff CSS 变量）

## 8. E1 — 大项目性能优化

**现状**：知识库条目多、故事列表长时可能卡顿，PRD 长文渲染偶尔延迟。

**方案**：针对三个场景优化，统一使用 IntersectionObserver 方案（`content-visibility: auto` 在低版本 macOS WKWebView 上兼容性不确定，作为增强而非主方案）。

### 8a. 知识库懒渲染

- Knowledge 页面的条目列表使用 IntersectionObserver + 条件渲染
- 离屏卡片渲染为固定高度占位符，进入视口时挂载真实内容
- 增强：检测 `CSS.supports("content-visibility", "auto")`，支持时额外添加 `content-visibility: auto` + `contain-intrinsic-size`

### 8b. 故事看板懒渲染

- Stories 页面的看板在故事数 > 20 时，只渲染可视区域内的卡片
- 使用 IntersectionObserver 按需挂载卡片内容（与 8a 统一模式）

### 8c. PRD 长文渲染优化

- PrdViewer 中 Mermaid 图表延迟渲染（IntersectionObserver 触发，滚动到视口内才初始化）
- Markdown 渲染结果缓存（相同内容不重复渲染）

## 9. E2 — 统一错误处理

**现状**：大部分异步操作仅 `console.error`，用户看不到错误信息。Tauri 命令失败时前端静默。

**方案**：建立统一的错误处理层，职责分离（API 层统一类型，UI 层翻译展示）。

### 错误分层

| 层级 | 处理方式 | 示例 |
|------|---------|------|
| **致命错误** | ErrorBoundary 全屏错误页 | React 渲染崩溃 |
| **操作错误** | Toast error + 操作按钮 | API 调用失败、文件读写失败 |
| **静默错误** | 仅日志 | 非关键 UI 更新失败 |

### API 错误包装

在 `tauri-api.ts` 中统一错误类型（**不格式化用户可见文案**，那是 UI 层职责）：

```typescript
export class AppError extends Error {
  constructor(
    public readonly cmd: string,
    public readonly rawMessage: string
  ) {
    super(rawMessage)
    this.name = "AppError"
  }
}

async function safeInvoke<T>(cmd: string, args?: Record<string, unknown>): Promise<T> {
  try {
    return await invoke<T>(cmd, args)
  } catch (err) {
    const message = typeof err === "string" ? err : String(err)
    const error = new AppError(cmd, message)
    logError(error)  // 写入本地日志
    throw error
  }
}
```

### 错误翻译（UI 层）

- 新建 `app/src/lib/error-messages.ts`：常见错误消息 → 用户友好中文的映射表
- 页面 catch handler 中调用 `toast(translateError(error), "error")`
- 未知错误降级为「操作失败，请重试」
- 后端返回的 Rust 错误保持原样传递，不在 API 层做翻译（保留条件判断能力）

### 错误日志

- 新建 `app/src/lib/error-log.ts`：
  - 写入 `localStorage`（key: `app-error-log`），最近 50 条
  - 单条日志限制 2000 字符截断
  - 包含：时间戳、命令名、错误消息、页面路径
- Settings 关于 Tab 中增加「错误日志」折叠区块，展示最近错误记录

**改动文件**：
- 修改：`app/src/lib/tauri-api.ts`（AppError 类 + safeInvoke 包装）
- 新建：`app/src/lib/error-messages.ts`（错误翻译映射表）
- 新建：`app/src/lib/error-log.ts`（本地错误日志）
- 修改：各页面的 catch handler（`console.error` → `toast(translateError(e), "error")`）
- 修改：`app/src/pages/settings/SettingsAbout.tsx`（错误日志查看）

## 10. E3 — E2E 测试框架

**现状**：测试覆盖为 0，无任何自动化测试。

**方案**：分两阶段搭建测试能力。

### Phase 1（本版本实现）：Vite Dev Server + Mock Invoke

Playwright 无法直连 Tauri WKWebView（macOS），第一阶段测试 Vite dev server 启动的 Web 版本：

- Playwright 连接 `http://localhost:5173`（Vite dev server）
- Mock 所有 `@tauri-apps/api` 的 `invoke` 调用（在 test setup 中注入 mock 层）
- 测试 UI 逻辑和交互流程，不测试 Tauri IPC 真实行为

**Mock 策略**：
- `app/tests/e2e/mocks/tauri-mock.ts`：mock `invoke`，按命令名返回预设数据
- AI 生成命令返回固定文本
- 文件操作返回模拟结果
- 项目 CRUD 操作内存态管理

**核心测试用例**（第一批）：

| 用例 | 覆盖路径 |
|------|---------|
| 创建项目 | Dashboard → 新建对话框 → 输入名称 → 创建成功 → 跳转 |
