# v0.2.5 迭代设计方案

## 概述

7 个功能点，覆盖 AI 质量深化、自定义能力、需求导入三个方向。优先级：AI 质量 > 自定义 > 集成。

> F1（Web 搜索）经审视确认已在 `claude_cli.rs` 中实现（research 阶段已返回 `WebSearch,WebFetch,Read`），从本版本移除。原 F8（飞书导出）在头脑风暴阶段已移除。编号保留不重编，避免混淆。

## 通用约定

沿用 v0.2.4 约定：
- 新功能必须处理空状态、加载态、错误态（具体要求见各功能的「边界状态」小节）
- 嵌入阶段页面时遵循 `PhaseShell` 布局结构
- 错误走 `safeInvoke` + `translateError` 统一处理
- 新组件文件名统一使用 kebab-case（如 `prd-score-panel.tsx`）
- 颜色必须使用 CSS 变量（`var(--success)` / `var(--destructive)` 等），禁止硬编码色值

### 公共基础设施（前置工作）

**1. 提取 `call_ai_non_streaming` 为公共函数**

当前 `call_ai_non_streaming` 定义在 `knowledge.rs` 中且非 `pub`，F2 需要复用。

- 将 `call_ai_non_streaming`、`call_ai_via_api`、`call_ai_via_cli` 提取到 `providers/ai_call.rs`（或 `providers/mod.rs` 新增子模块）
- 接口签名：`pub async fn call_ai_non_streaming(config_dir: &str, prompt: &str) -> Result<String, String>`
- `knowledge.rs` 中的 `extract_knowledge_candidates` 改为调用新的公共函数
- `score_prd`（F2）直接调用公共函数

**2. DB 迁移版本管理**

当前 `init_db` 使用 `ALTER TABLE ... ADD COLUMN` + 忽略全部错误的方式，无法区分「列已存在」和真正故障。

- 新增 `schema_version` 表：`CREATE TABLE IF NOT EXISTS schema_version (version INTEGER NOT NULL)`
- 启动时读取版本号，按版本号执行增量迁移
- F5 的 `project_type` 字段作为 migration v2 执行
- 旧版数据库（无 `schema_version` 表）视为 v1，先补建表再执行后续迁移

**3. System prompt 体积保护**

`build_system_prompt` 已拼接大量内容（技能文件、知识库、上下文文件、参考文件、设计规范等），F5/F7 还会追加更多。

- `build_system_prompt` 末尾新增总长度检查
- 超过 300K 字符（约 80K tokens）时，按优先级裁剪：知识库推荐 → 参考文件 → 旧的上下文文件
- 超限时通过 `stream_warning` 事件通知前端，前端 Toast warning 显示

**4. 前端类型定义**

- `phase-meta.ts` 新增 `REQUIRED_PHASES = ["requirement", "prd"] as const`
- `tauri-api.ts` 新增 `type ProjectType = "general" | "to-b" | "to-c" | "internal"` 联合类型
- `tauri-api.ts` 新增 `PROJECT_TYPE_META` 常量（图标、名称、描述），统一管理
- `ProjectSummary` 和 `ProjectDetail` 均增加 `projectType?: ProjectType` 可选字段（兼容已有项目）

## 2. F2 — PRD 质量自检评分

**现状**：PRD 生成后用户不知道质量如何，全凭主观判断。

**方案**：PRD 生成完成后，自动运行一次非流式 AI 评估，输出 5 维度评分卡。

**评估维度**（每项 1-5 分）：

| 维度 | 说明 |
|------|------|
| 完整性 | 是否覆盖了目标用户、核心功能、非功能需求、验收标准 |
| 清晰度 | 描述是否无歧义，开发可直接理解 |
| 可执行性 | 功能描述是否足够详细，能否直接转化为开发任务 |
| 一致性 | 各章节之间是否矛盾（如功能列表与用户故事是否对应） |
| 边界定义 | 是否明确了「不做什么」、异常处理、边界条件 |

**后端实现**：

- 新增 `score_prd` 命令，放在 `commands/files.rs`（与 PRD 导出同模块）
- 调用公共基础设施的 `call_ai_non_streaming`（见前置工作 1）
- **Mutex 安全**：先在 `{}` 块内获取 lock 读取 output_dir，释放后再调用 AI（遵循 `extract_knowledge_candidates` 同样的模式）
- 输入：PRD 全文，使用 `truncate_to_chars` 函数截断到 8000 字符（复用 `knowledge.rs` 现有函数）
- 输出 JSON：`{ dimensions: [{ name, score, comment, suggestion }], totalScore }`
- 评分结果缓存到 `05-prd/quality-score.json`，包含 PRD 内容 hash（`content_hash` 字段）
- **缓存失效**：读取缓存时比对当前 PRD 内容 hash，不一致则标记为 stale

**前端交互**：

- PRD 生成完成后自动调用 `score_prd`
- 标题区域显示总分 badge（如 `4.2/5`），颜色：≥4.0 `var(--success)`、2.5-3.9 `var(--warning)`、<2.5 `var(--destructive)`
- **默认折叠**，不自动展开详情面板。用户主动点击 badge 才展开（`fadeInUp` 动画）
- 每个维度一行：维度名 + 分数条 + 评语 + 改进建议
- 改进建议采用多选模式：每条左侧复选框，底部统一「发送已选建议到 AI Assist」按钮（避免重复按钮过多）
- 内容变更后 badge 显示为 stale 态（灰色 + 「重新评分」文字），用户点击触发重评

**边界状态**：
- 加载态：badge 位置显示 Skeleton 占位 + 脉冲动画「评分中...」
- 错误态：badge 显示「评分失败」灰色文字，点击可重试
- 空态（无 PRD）：不显示 badge

**改动文件**：
- 前置：提取 `call_ai_non_streaming` 到公共模块
- 后端新增：`score_prd` 命令（`commands/files.rs`）
- 新建：`app/src/components/prd-score-panel.tsx`
- 修改：`app/src/pages/project/Prd.tsx`（评分触发 + badge + 详情面板）
- 修改：`app/src/lib/tauri-api.ts`（新增 `scorePrd` API + `PrdScore` 类型）

## 3. F3 — AI 对话式 PRD 编辑

**现状**：AI Assist 是单次指令模式，每次修改重写全文，无法多轮对话细调。

**方案**：AI Assist 区域升级为对话面板，支持多轮连续编辑，每轮修改以变更块级粒度接受/拒绝。

**交互流程**：

```
PRD 页面（已有内容）
  → 底部固定的单行输入条（默认折叠态）
  → 用户点击展开按钮 → 向上推开为对话面板（最大高度不超视口 40%）
  → 输入修改指令 → AI 返回修改后全文
  → 前端 diffLines 计算差异，按变更块展示（非行级）
  → 每个变更块可接受/拒绝，提供「全部接受」「全部拒绝」快捷按钮
  → 接受的修改应用到 PRD，对话历史保留
  → 用户可继续输入下一轮修改指令
```

**关键设计决策**：

- **Diff 粒度**：变更块级（hunk-level），与 `diffLines` 的 `Change` 粒度一致，不做行级。每个 Change 可接受或拒绝
- **Diff 由前端计算**：AI 只返回修改后全文，前端用 `diffLines` 对比新旧文本生成 diff。不要求 AI 输出 diff 格式（不可靠）
- **AI 响应格式**：使用自定义标记分隔 `---SUMMARY---\n{摘要}\n---FULL_TEXT---\n{全文}`，比 Markdown 标题更鲁棒
- **降级处理**：找不到分隔标记时，将整个输出视为修改后全文，Toast warning「差异解析异常，已切换为整体替换模式」，给用户「查看完整替换预览」和「取消」两个选项

**上下文窗口管理**：

- 仅首轮携带完整 PRD 全文
- 后续轮次只发上一轮 diff 摘要（前 500 字符）+ 新指令，不重复发全文
- 对话历史滑动窗口：prompt 中最多带最近 3 轮摘要
- 持久化到 `brainstorm_messages` 表（复用现有表，phase 设为 `"prd-assist"`），不另建 JSON 文件

**面板布局**：

- 默认折叠为底部固定的单行输入条（类似聊天窗口）
- 点击展开后向上推开，主内容区同步收缩。展开/折叠使用 `ease-decelerate` 200ms 过渡
- 面板内显示最近 20 轮对话历史，更早的折叠为「查看更多」
- 面板标题栏更多菜单中提供「清除对话历史」
- 项目设置 Drawer 打开时，对话面板自动折叠

**PrdDiffViewer 改造**：

- 新增 `interactive` prop（默认 false，保持纯展示兼容）
- `interactive=true` 时每个 Change 块右侧显示接受/拒绝按钮
- 内部维护 `accepted: Set<number>` 状态
- 新增 `onResolve(resolvedText: string)` 回调，父组件调用获取合成结果

**边界状态**：
- 加载态：AI 生成中显示 `thinkingPulse` 动画 + 「AI 正在修改...」
- 错误态：对话面板内 inline 显示错误 + 重试按钮
- 空态（无对话历史）：面板内显示引导文字「输入修改指令，如"把权限配置写详细些"」

**改动文件**：
- 新建：`app/src/components/prd-assist-panel.tsx`（对话面板组件）
- 修改：`app/src/components/PrdDiffViewer.tsx`（新增 interactive 模式 + hunk 级接受/拒绝）
- 修改：`app/src/pages/project/Prd.tsx`（替换当前 AI Assist input 区域）

## 4. F4 — 评审意见关联 PRD 章节 + 一键采纳

**现状**：评审报告和 PRD 是独立文档，评审指出的问题需要用户手动对照修改。

**方案**：评审意见按 PRD 章节分组，支持一键发送到 PRD 页面的 AI Assist 修改。

**依赖关系**：F4 与 F3 无硬依赖。F4 的「采纳」功能使用现有 AI Assist 能力（已存在）。如果 F3 已完成，采纳会自动受益于对话面板；如果 F3 未完成，采纳降级为填入现有单行 input。两者可并行开发。

**后端 prompt 约束**：

- Review 阶段 prompt 新增指令：提取 PRD 章节标题列表（从 `05-prd/05-PRD-v{N}.0.md` 中提取 `## ` 开头的行），在 prompt 中显式枚举可选章节名称
- 要求 AI 输出格式：每条意见标注 `[章节：{从枚举列表中选择}]`
- 这比让 AI 自由发挥章节名称更稳定

**前端 Review 页面**：

- 新增「按章节」视图模式，与现有「按角色」Tab 通过 SegmentedControl 切换
- 「按章节」视图：正则匹配 `[章节：xxx]` 标记分组，每章节一个折叠区块
- 未标注章节的意见归入「全局建议」分组
- 识别不到格式时，「按章节」Tab disabled + tooltip 提示「评审报告格式不支持章节分组」

**采纳机制**：

- 每条意见右侧「采纳」按钮
- **跨页面状态传递**：使用 `sessionStorage`（key: `review-adoption-queue`）存储待采纳意见列表，不用 URL 参数（避免长度限制）
- 点击「采纳」→ 意见写入 sessionStorage → 标记为 `adopted`（绿色 check + 删除线）→ 可选跳转（「采纳」旁增加「采纳并前往 PRD」链接）
- PRD 页面打开时读取 sessionStorage，如有待处理意见则自动填入 AI Assist 并清除

**意见状态**：

- 存储在 `08-review-status.json`：`{ [意见内容摘要hash]: "pending" | "adopted" | "dismissed" | "deferred" }`
- 新增 `deferred`（暂缓）状态，灰色标记
- 读取时做弱校验：存储的 hash 与当前意见内容比对，不一致则忽略旧状态
- PRD 标题栏显示待处理意见数 badge（`var(--warning)` 色，仅 pending 计数）

**边界状态**：
- 空态（无评审报告）：「按章节」Tab 内显示「尚无评审意见，完成评审后将在此展示」
- 加载态：章节分组区域 Skeleton
- 错误态：解析失败 Toast + 降级为纯 Markdown 渲染

**改动文件**：
- 修改：`app/src-tauri/src/commands/stream.rs`（review 阶段 prompt 枚举章节名称）
- 修改：`app/src/pages/project/Review.tsx`（SegmentedControl 视图切换 + 章节分组展示 + 采纳按钮）
- 新建：`app/src/components/review-grouped-view.tsx`（按章节分组组件）
- 修改：`app/src/pages/project/Prd.tsx`（sessionStorage 读取 + 待处理意见 badge）

## 5. F5 — 项目类型模板

**现状**：所有项目使用同一套 prompt，不同领域项目生成内容风格雷同。

**方案**：新建项目时可选择项目类型，每种类型关联不同的阶段 prompt 补充指令。

**预置类型**：

| 类型 | 枚举值 | prompt 侧重 |
|------|--------|-------------|
| **通用**（默认） | `general` | 当前行为，不追加额外指令 |
| **To B 企业应用** | `to-b` | 强调角色权限矩阵、审批流程图、与现有系统的集成方案、数据迁移 |
| **To C 用户产品** | `to-c` | 强调用户增长漏斗、核心指标（DAU/留存/转化）、竞品体验对比、AB 测试方案 |
| **内部工具** | `internal` | 精简 PRD 结构、跳过市场分析、重点放在效率提升指标和操作流程优化 |

**后端实现**：

- `projects` 表新增 `project_type TEXT DEFAULT 'general'`（通过 DB 迁移版本管理，见前置工作 2）
- `create_project` 命令新增 `projectType` 参数（`CreateProjectArgs` 增加字段）
- 模板存储：`templates/project-types/{type}/prompt-overrides.json`
- `build_system_prompt`：读取项目类型 → 加载对应 `prompt-overrides.json` → 用分隔标记包裹后追加：`\n\n<!-- project-type-override -->\n{内容}\n<!-- /project-type-override -->`
- **prompt 注入防护**：模板内容限制最大 4000 字符，超出截断
- 预置模板随应用打包（`resources/templates/project-types/`）
- **模板同步策略**：模板 JSON 中包含 `version` 字段，启动时比对版本，只在版本更高时覆盖用户目录的文件。用户修改过的文件（mtime 晚于上次同步时间）不覆盖

**前端交互**：

- 新建项目对话框：项目名下方 4 张小卡片选择类型
- 卡片样式：选中态 `--accent-color` 描边 + `--accent-light` 底色，未选中 `--border` 描边 + `--card` 底色
- 每张卡片：图标 + 类型名 + 一句话描述（明确说明预设了哪些差异）
- 数据来源：`PROJECT_TYPE_META` 常量（见前置工作 4），前后端共享枚举值
- 项目设置面板可切换类型（弹窗确认）

**改动文件**：
- 修改：`app/src-tauri/src/db.rs`（migration v2 + schema_version 表）
- 修改：`app/src-tauri/src/commands/projects.rs`（`CreateProjectArgs` + `get_project` 支持 project_type）
- 修改：`app/src-tauri/src/commands/stream.rs`（`build_system_prompt` 读取类型模板）
- 新建：`app/src-tauri/resources/templates/project-types/` 下 3 个 JSON 文件
- 修改：`app/src/components/new-project-dialog.tsx`（类型选择 UI）
- 修改：`app/src/lib/tauri-api.ts`（`ProjectType` + `PROJECT_TYPE_META` + 类型扩展）
- 修改：`app/src/lib/phase-meta.ts`（新增 `REQUIRED_PHASES`）

## 6. F6 — 自定义阶段开关

**现状**：v0.2.4 的阶段跳过是 AI 推荐后用户确认，缺乏主动控制。

**方案**：项目设置中用户可手动开关各阶段。

**交互设计**：

- **新建项目对话框**：底部仅做轻量预览（文字说明「包含 9 个阶段，创建后可在项目设置中自定义」），不在新建流程中展示完整阶段开关（降低认知负荷）
- 选择项目类型为「内部工具」时自动关闭 `research` 和 `analytics`，新建后在项目设置中可改回

**项目设置 Drawer**：

- 侧边栏底部新增齿轮图标按钮「项目设置」
- 点击弹出 Drawer（宽度 480px），包含两个 Tab：「阶段管理」+「Prompt 定制」（F7）
- 阶段管理 Tab：9 个阶段的 Switch 列表
- 每个 Switch 旁有一句话说明（13px `--text-secondary`），解释该阶段做什么
- `requirement` 和 `prd` 强制开启：disabled + `opacity-50` + 锁图标 + tooltip「核心阶段，不可关闭」
- 已完成阶段：disabled + tooltip「已完成，不可回退」
- 切换即时生效：关闭 → 调用 `skip_phases`，打开 → 调用 `unskip_phase`（复用 v0.2.4 命令）
- Drawer 打开时底部 AI Assist 面板自动折叠

**后端校验**：

- `skip_phases` 命令增加校验：拒绝跳过 `requirement` 和 `prd`
- `unskip_phase` 命令增加校验：检查该阶段之后是否有已完成的阶段，如果有则拒绝恢复（返回错误信息）

**事件通信**：

- Switch 切换后触发 `project-phase-updated` 自定义事件
- `PhaseShell` 已监听此事件，自动刷新 skip 状态

**边界状态**：
- Drawer 加载态：Switch 列表 Skeleton
- 操作失败：Toast error

**改动文件**：
- 新建：`app/src/components/project-settings-drawer.tsx`（项目设置面板）
- 修改：`app/src/components/layout/Sidebar.tsx`（齿轮按钮入口）
- 修改：`app/src/components/new-project-dialog.tsx`（轻量预览文字 + 内部工具默认关闭阶段）
- 修改：`app/src-tauri/src/commands/projects.rs`（`skip_phases`/`unskip_phase` 增加校验）

## 7. F7 — 自定义 Prompt 模板编辑器

**现状**：阶段 prompt 写在技能文件里，用户无法调整。

**方案**：项目级别的 prompt 补充指令编辑器。

**交互设计**：

- 位于 ProjectSettingsDrawer 的「Prompt 定制」Tab
- 列出所有已启用的阶段，手风琴模式（同一时间只展开一个）
- 每个阶段：标题行（阶段名 + 简短默认 prompt 摘要前 50 字 + 省略号）
- 展开后：
  - 上方可折叠预览区块（灰色底色 `--secondary`，最大高度 120px 可滚动）显示默认 prompt 完整内容
  - 下方多行 textarea（最小高度 120px，可拖拽调整），placeholder「输入补充指令，将追加到默认 prompt 之后」
  - textarea 右上角有该阶段独立的「重置」ghost 按钮
- 支持占位符变量：`{{项目名}}` `{{项目类型}}` `{{行业}}`，生成时自动替换
- 底部全局「重置为默认」按钮需二次确认 Dialog

**存储**：

- 存储在 DB 中新增 `project_prompt_overrides` 表（`project_id TEXT, phase TEXT, prompt_text TEXT, PRIMARY KEY(project_id, phase)`）
- 不存在项目 output 目录中，避免导出/导入时产生副作用

## 8. F9 — 需求导入（拖放/文件/剪贴板）

**现状**：需求只能在 Requirement 页面手动输入，实际需求来自飞书消息、邮件、文档。

**方案**：Requirement 页面支持拖放文件、文件选择、富文本粘贴三种导入方式。

**拖放导入**：

- 需求输入区域（TipTap 编辑器外围）增加 `onDragOver` / `onDrop` 事件监听
- 支持文件类型：`.txt`、`.md`（前端直接 `FileReader` 读取）、`.docx`（调用后端提取）
- **空/非空两种显示状态**：
  - 页面为空时：拖放区域作为空状态占位，居中显示（蓝色虚线 + 图标 + 「拖放文件到此处，或点击导入」文案）
  - 页面已有内容时：拖放区域隐藏，仅在用户拖拽文件进入窗口时以覆盖层形式出现（半透明蓝色 + 虚线边框动画），释放后消失
- 不支持的文件格式拖入后 Toast error「不支持该文件格式，请使用 .md .txt .docx 文件」
- 拖放区域增加 `aria-label="拖拽或点击选择文件导入需求"` + `role="region"`

**文件选择导入**：

- 输入区域上方工具栏新增「导入」按钮（Upload 图标），支持键盘聚焦和 Enter/Space 激活
- 点击调用 Tauri `open` dialog，过滤 `.txt .md .docx`

**DOCX 文本提取**：

- **技术选型：Rust 原生**（zip + quick-xml），不依赖 Python
- 后端新增 `extract_docx_text` 命令：解压 DOCX → 读取 `word/document.xml` → 提取 `<w:t>` 文本节点
- **安全防护**：
  - zip entry 路径规范化检查，拒绝包含 `..` 或绝对路径的条目
  - XML 解析使用 `quick-xml`（已在 Tauri 依赖树中），禁用外部实体
  - 输入文件大小上限 50MB，解压后内容总大小上限 100MB
- 只提取纯文本，不处理样式、图片等复杂格式

**富文本粘贴**：

- 监听 TipTap 编辑器的 paste 事件
- 检测 `clipboardData.types` 包含 `text/html` 时，提取 HTML 并转为 Markdown
- HTML → Markdown 简单映射：`<p>` → 段落、`<li>` → `- `、`<h1-6>` → `#`、`<strong>` → `**`、`<a>` → `[text](href)`
- 如果检测到格式损失（如复杂表格），Toast info「部分复杂格式可能需要手动调整」
- 如果只有 `text/plain`，走默认粘贴行为

**导入后处理**：

- 追加到末尾，前面自动插入分隔线 + 导入时间标注：`\n---\n*以下内容导入于 {时间}*\n`
- Toast 提示「已导入 {N} 字内容」，通过 `aria-live="polite"` 通知屏幕阅读器
- 工具栏「导入」按钮下拉菜单中提供「替换导入」选项作为备选

**边界状态**：
- 加载态（DOCX 解析中）：Toast + 按钮 disabled + 「导入中...」
- 错误态：Toast error + 具体错误信息
- 空文件：Toast warning「文件内容为空」

**改动文件**：
- 修改：`app/src/pages/project/Requirement.tsx`（拖放区域 + 导入按钮 + 粘贴处理）
- 后端新增：`extract_docx_text` 命令（`commands/files.rs`）
- 修改：`app/src-tauri/Cargo.toml`（添加 `quick-xml` 依赖，如未在依赖树中）
- 修改：`app/src/lib/tauri-api.ts`（`extractDocxText` API）
