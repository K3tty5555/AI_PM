一、简介与快速开始

1.1 什么是 AI_PM？

AI_PM 是一个模仿产品经理工作流的 Claude Code 技能套件。它能够将简短的需求描述转化为完整、专业的产品需求文档（PRD）和可交互原型。

二、用户分层模式

AI_PM v1.4 引入了三层用户体系，根据你的经验水平提供个性化体验：

层级 特征 交互特点 默认模式
新手 产品经验 < 2年 详细引导、解释术语、示例丰富 标准+详细引导
专业 资深 PM，有方法论 标准流程、可自定义、高效产出 标准模式

三、四种工作流详解

3.1 快速模式（⚡ Quick Mode）

适用场景：简单工具类 App、已有清晰思路、个人项目/原型验证、时间紧迫

流程：需求澄清 (5min) → PRD生成 (8min) → 原型 (2min)

/ai-pm quick "需求描述"

3.2 标准模式（📊 Standard Mode）

适用场景：常规产品项目、需要完整分析、团队协作

流程：完整 Phase 1-6，预计 45 分钟

3.3 深度模式（🔬 Deep Mode）

适用场景：复杂企业级系统、战略级产品、需要多方评审

流程：Phase 0-7 + 多轮评审，预计 2 小时

3.4 自适应模式（🎯 Adaptive Mode）

AI 根据需求描述长度、技术关键词密度、业务复杂度评估自动选择模式

五、需求评审详解

5.1 什么时候用？

  • PRD 完成后，想提前发现问题
  • 重要项目，需要多方把关
  • 团队协作，需要达成共识
  • 学习提升，想了解专业视角

5.2 八大评审角色

角色 关注维度 典型问题
👨‍💼 研发总监 技术可行性、开发成本、工期评估 技术方案不明确、工期不合理
🏗️ 资深架构师 系统架构、技术选型、扩展性 缺少缓存策略、架构不合理
📊 产品总监 产品定位、商业价值、需求优先级 P0 过多、用户分层不细
🎨 设计总监 用户体验、交互设计、视觉规范 缺少空状态、流程不顺畅
📅 项目经理 项目排期、风险管理、资源协调 里程碑过密、依赖未明确
🧪 QA 总监 可测试性、验收标准、边界覆盖 缺少验收标准、边界未覆盖
🔒 安全合规官 数据安全、隐私保护、合规审查 敏感信息未脱敏、权限过宽
📈 运营增长官 冷启动、运营工具、数据埋点 缺少冷启动方案、运营后台弱

5.3 问题分级

级别 说明 处理要求 典型场景
Critical 影响核心功能或存在重大风险 必须修改，阻塞发布 安全漏洞、合规风险、架构缺陷
Major 影响用户体验或存在明显缺陷 建议修改，影响满意度 体验断层、测试覆盖不足、项目风险
Minor 细节优化或改进建议 可选修改，锦上添花 文案优化、交互微调

5.4 评审流程

1. 触发评审
   /ai-pm review

2. AI 自动分配八个角色并行评审
   [研发总监]   评审中... ✓
   [架构师]     评审中... ✓
   [产品总监]   评审中... ✓
   [设计总监]   评审中... ✓
   [项目经理]   评审中... ✓
   [QA总监]     评审中... ✓
   [安全合规官] 评审中... ✓
   [运营增长官] 评审中... ✓

3. 生成评审报告 08-review-report-v1.md
   - 按严重程度汇总问题
   - 给出修改分工建议

4. 用户确认修改范围
   - 全部修改 / 核心修改 / 最小修改 / 安全优先

5. AI 自动修改 PRD（在原文档直接更新）
   - 在 05-PRD-v1.0.md 的修订日志中增加 v1.1 记录
   - 直接在原文档中修改内容
   - 不创建新文件版本，保持文档唯一性

6. 再次评审（可选）/ai-pm review --round=2
⚠️
v2.0 更新：评审后修改方式优化
之前：创建 05-PRD-v1.1.md 新文件
现在：直接在 05-PRD-v1.0.md 中更新，通过修订日志追踪变更
好处：保持文档唯一性，避免版本文件冗余

5.5 场景化评审配置

💡
不同场景可启用不同角色组合：
• 创业团队：产品+研发+设计（轻量级）
• 企业级项目：八角色全启（全量评审）
• 内部工具：产品+研发+QA（简化评审）
• 安全敏感：增加安全合规专项评审

七、最佳实践

7.1 选择合适的模式

✓
场景 1：简单工具 App → 快速模式
场景 2：常规产品迭代 → 标准模式

  • 多轮评审：重要项目建议至少两轮评审

八、高级自定义

AI_PM 支持深度自定义，让生成的 PRD 和原型更符合你的企业和个人风格。

8.1 模板系统（templates/）

templates/ 目录用于存放全局模板，所有项目共享：

templates/
├── writing-styles/         # 写作风格库（PRD 文档风格）
│   ├── default/            # 默认风格
│   └── [自定义风格]/       # 你创建的风格
├── ui-specs/               # UI 规范库（设计规范）
│   ├── example-enterprise/ # 企业规范示例
│   └── [自定义规范]/       # 你上传的规范
├── prd-templates/          # PRD 模板
└── 01-config/              # 配置模板
    └── reference-config.md # 参考资源配置模板

8.2 PRD 写作风格管理

分析你历史 PRD 的写作风格（章节结构、用词习惯、详细程度等），后续生成时自动应用，保持文档风格一致性。

💡
什么是写作风格？ 包括：章节结构偏好、用词习惯（技术/口语）、详细程度、表格使用频率、示例丰富度等

创建写作风格

# 分析你现有的 PRD 文件，提取写作风格特征
/ai-pm writing-style analyze ~/Documents/我的PRD.md

# 命名并保存风格
# 风格文件将保存到 templates/writing-styles/[风格名]/

应用写作风格

# 生成 PRD 时会自动提示选择风格
/ai-pm prd

# 选择：default / enterprise-standard / 你创建的风格

管理写作风格

/ai-pm writing-style list     # 查看所有可用风格
/ai-pm writing-style switch  # 切换默认风格

8.3 UI 规范管理

上传企业的 UI 设计规范（颜色、字体、间距、组件），原型生成时自动应用。

上传 UI 规范

# 创建规范目录
mkdir -p templates/ui-specs/my-company

# 放入设计资源：
# - design-tokens.json    # 设计令牌（颜色、字体、间距）
# - components/           # 组件截图或描述
# - guidelines.md         # 设计准则文档
# - brand-assets/         # Logo、图标等

自动生成设计令牌（推荐）

# 上传设计稿图片或 PDF
/ai-pm ui-spec upload my-company

# AI 自动解析：
# - 主色/辅助色/功能色
# - 字体层级
# - 间距系统
# - 圆角规范

应用 UI 规范

# 生成原型时会自动提示选择 UI 规范
/ai-pm prototype

# 选择：example-enterprise / my-company

UI 规范优先级

  1. 项目级 UI 规范（项目目录下的 ui-spec/）
  2. 全局 UI 规范（templates/ui-specs/）
  3. 默认规范（Apple HIG）

8.4 自定义 PRD 模板

如果你有特定的 PRD 格式要求（如公司模板），可以创建自定义模板：

# 1. 创建模板目录
mkdir -p templates/prd-templates/my-company

# 2. 放入模板文件
# - template.md           # PRD 模板主体
# - style-config.json     # 风格配置

# 3. 生成时选择模板
/ai-pm prd
# → 选择模板：default / my-company
✓
最佳实践：先完成 2-3 个项目，积累满意的 PRD 后，再提取写作风格并上传 UI 规范，实现真正的个性化

八、v2.0 核心优化详解

v2.0 版本基于用户反馈进行了三项核心优化，大幅提升用户体验：

8.1 项目类型确认（首要问题）

优化前：默认按 0-1 全新开发处理，不适合迭代项目

优化后：首先询问项目类型，三种模式不同策略

项目类型 询问重点 输出侧重
0-1 全新开发 目标用户、核心痛点、竞品分析 完整需求分析、用户画像、市场定位
基于已有项目迭代 现有系统问题、用户反馈、改进点 问题分析、改进方案、兼容过渡
参考竞品/对标 竞品优势、差异化点、超越策略 竞品对比、差异化功能、突破点

8.2 参考资源收集（关键优化）

优化前：被动等待用户提供参考，经常遗漏

优化后：主动询问并自动抓取，形成 Phase 0

Phase 0: 参考资源收集（v2.0 新增）
    ↓
💬 主动询问：你是否有以下参考资源？

📱 现有系统（如果是迭代项目）：
   • 现网 URL 和测试账号密码
   • 现有系统截图或录屏
   • 现有功能清单或 PRD

🎯 竞品参考：
   • 竞品网址（可公开访问的）
   • 竞品截图/UI/UE 设计稿
   • 你关注的具体功能点

🎨 设计参考：
   • UI 设计稿、品牌规范
   • 参考网站（你喜欢的风格）
   • 设计系统/组件库文档
    ↓
如果用户提供了资源：
   • URL → 使用 playwright-cli 自动抓取网页
   • 截图 → 保存到 07-references/images/
   • 账号 → 登录后抓取内部系统
    ↓
生成 07-references/reference-config.md（配置记录）
生成 00-reference-analysis.md（分析报告）
💡
07-references/ 目录结构：
• reference-config.md - URL、账号密码等配置
• session-auth.json - 保存的登录会话
• images/ - 参考图片（UI截图、竞品截图）

8.3 评审后修改优化（文档管理）

优化前：创建新文件 05-PRD-v1.1.md，文件数量不断增加

优化后：直接在原文档更新，通过修订日志追踪变更

对比项 v1.x 方式 v2.0 方式
修改方式 创建 05-PRD-v1.1.md 新文件 在 05-PRD-v1.0.md 中直接修改
版本追踪 多个文件 修订日志 + Git 历史
文件数量 不断增加（v1.0, v1.1, v1.2...） 保持唯一（始终 v1.0）
备份文件 生成 .bak 文件 不生成（Git 已足够）

v2.0 修改流程：

1. 读取 05-PRD-v1.0.md
2. 在修订日志中增加新记录：
   | 2026-03-01 | v1.1 | 修复30个问题（2C+12M+16m）| 全模块 | AI_PM | 详见第9章 |
3. 直接在原文档中修改内容
4. 新增"评审问题修复记录"章节（如第9章）
5. 可选生成 08-review-changes.md（详细修改说明）
6. 不创建 .bak 备份文件
⚠️
注意事项：
• 始终只保留 05-PRD-v1.0.md 一个文件

九、常见问题

Q1: 新手从哪里开始？

建议：先选择「新手模式」体验一次完整流程 → 阅读生成的文档 → 尝试用「快速模式」做个简单需求 → 熟悉后切换到「专业模式」

Q2: 快速模式和标准模式有什么区别？

对比项 快速模式 标准模式
时间 15 分钟 45 分钟
流程 3 个阶段 6 个阶段
分析深度 基础 深入
适用场景 简单需求 常规项目

Q3: 需求评审必须做吗？

不是必须的，但强烈推荐用于：复杂项目、团队协作、学习提升、高风险项目。简单个人项目可以跳过。

Q4: 如何提升 PRD 质量？

  • 提供详细的需求描述
  • 选择深度模式 + 多轮评审
  • 上传参考资源
  • 创建自定义风格
  • 使用需求评审功能
