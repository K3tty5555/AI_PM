# 数据分析前必须先澄清业务目标，不要把假设当方向

## 问题场景

拿到学校考试+作业双端数据，基于「买了作业产品 → 会减少手阅」的假设开始分析，生成了一份以「手阅转化」为核心命题的报告。写了一半，用户指出：方向就错了，实际目标是「考试和作业的产品融合」，手阅转化只是一个未经验证的猜想。

整份报告需要推翻重写。

## 怎么躲

**开始分析前，先问清楚业务目标**：

```
1. 这份分析是为了回答什么问题？
2. 有没有已经形成的假设需要验证？
3. 结论会用来做什么决策？（产品路线图 / 销售话术 / 运营策略）
4. 受众是谁？他们最关注哪个维度？
```

本次教训：「手阅率分析」是工具，「融合产品机会」才是目的。分析框架一旦选错，所有数据解读都会走偏。

## 与数据假设验证的区别

- **假设验证**（正确用法）：明确说「我们要验证 X」，分析结果可以是「假设成立」或「假设不成立」
- **假设当方向**（本次问题）：把「买作业会减少手阅」当成已知结论，只找支持它的数据

本次数据实际结果：多数学科引入作业后手阅率反而上升，说明假设不成立。但这个「不成立」本身也是有价值的洞察，而不是分析失败。

## 验证数据

本次返工重写了完整报告和仪表盘，耗时约 1 个完整会话。

## 适用场景

所有数据洞察项目，尤其是分析周期长、输出物多的情况（报告 + 仪表盘 + SOP）。
