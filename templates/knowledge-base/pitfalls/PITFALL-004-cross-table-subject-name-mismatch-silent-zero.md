# 跨表学科命名不一致会让整个学科数据静默消失，误判为产品空白

## 问题场景

考试表和作业表来自不同系统，同一学科命名不统一：

| 表 | 实际存储的学科名 |
|----|----------------|
| 考试明细 | 思想政治、政治、道德与法治（三种混用） |
| 作业明细 | 政治（统一） |

直接用学科字段做统计时，`subj_hw['政治']` 能匹配，但 `subj_hw['思想政治']` 和 `subj_hw['道德与法治']` 的数据落进了另一个 key，统计结果显示政治作业 = 0。

于是生成了一条"产品空白"洞察：**两省合计 39,162 场政治考试，作业条数为零**——听起来像重大机会，实际是统计 bug。

修复归一化后：湖北 803 条 / 江苏 316 条。虽然仍属少量，但结论性质完全不同。

## 怎么躲

**Step 0 必做：打印各表的学科名称分布**

```python
from collections import Counter
exam_subjs = Counter(r[he['学科']] for r in rows_e)
hw_subjs   = Counter(r[hh['学科']] for r in rows_h)
print('考试学科:', exam_subjs.most_common(15))
print('作业学科:', hw_subjs.most_common(15))
```

肉眼对比两表的 key，发现不一致后写归一化函数：

```python
def normalize_subj(s):
    if s in ('思想政治', '道德与法治'):
        return '政治'
    return s
```

**在所有处理两张表的循环里都调用归一化**，不要只处理考试表漏掉作业表。

## 黄金规则

**某个重要指标为 0 时，先排查命名，再下产品结论。**

"零" 有两种来源：
1. 真实业务空白（产品没有覆盖）
2. 命名 mismatch 导致的统计 bug

第 1 种是洞察，第 2 种是 bug。两者看起来一模一样，必须用 Step 0 区分。

## 验证数据

政治作业修复前后：
- 修复前：湖北 0 条 / 江苏 0 条（错误）
- 修复后：湖北 803 条 / 江苏 316 条（正确）

## 适用场景

任何多表关联分析，尤其是：
- 同一系统不同业务线的数据表（考试 vs 作业）
- 跨系统导出数据
- 包含"思想政治/道德与法治/政治"的学科字段
