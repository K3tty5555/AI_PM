# AI-PM 整体使用经验：效率陷阱与最佳实践

## 1. 有现成文档时，不要走完整访谈流程

**问题**：用户已有产品说明书/引擎输出定义/竞品截图，但 AI-PM 仍然走需求访谈，导致反复确认已知信息，浪费 1-2 轮来回。

**正确做法**：
```
已有需求文档？→ 放入 07-references/ → 直接读 → 跳过 Phase 1 访谈
```

ai辅批项目中，用户提供了「高中语文作文批改报告输出定义.pdf」和「考试阅卷 AI批改助手 V1.pdf」，读完两份文档后 Phase 1-4 可以在一次响应内完成，不需要多轮对话。

**关键**：AI-PM 的价值在于整合信息、生成结构化产出，而不是从头挖掘用户脑子里的信息。

---

## 2. 写 PRD 前必须先读模板，不能凭记忆

**问题**：用户有专用 feishu-template.md，但 AI 凭印象写了不同结构的 PRD，需要全部返工。

**正确做法**：
1. 进入 PRD 阶段**第一件事**：读 `templates/prd-styles/default/feishu-template.md`
2. 字段名、字段顺序、行结构 100% 以模板为准
3. SKILL.md 里的任何格式描述优先级低于模板文件

---

## 3. Mermaid 流程图不能静默降级为代码块

**问题**：PRD 要求必须用 Mermaid 流程图，渲染失败后 AI 静默改成了代码块，用户没意识到。

**正确做法**：
- 流程图渲染失败 → 修复渲染，不能降级
- 降级需要明确告知用户，并给出原因

---

## 4. 引擎/后端已有的参数不要做成用户配置项

**问题**（通用坑，AI辅批项目中触发）：PRD 初稿把「习作要求」和「扣分规则」做成了用户在配置页填写的项。实际上这两项是引擎内置规则。

**怎么躲**：
```
写功能前先问清：这个参数是用户决定、系统自动传参，还是引擎黑盒处理？
```
三类参数的处理方式完全不同，混淆会导致要么配置页过于复杂，要么遗漏关键控制点。

---

## 5. 会话中断后的恢复规范

**问题**：会话中断后用 `/ai-pm continue` 时，AI 可能读取了过期的需求草稿，基于旧信息继续写。

**正确做法**：
1. 恢复后先 `ls` 看项目目录所有文件的修改时间
2. 读最新版本文件，不假设内存中残留的内容是最新的
3. 如果 PRD 已有修订历史（如 v1.1 修订说明），以修订说明为准，不以 SKILL.md 里的模板为准

---

## 6. 一次只问最重要的一个问题

**问题**：AI-PM 把所有澄清问题一次打出来，用户回答了其中一个，其他问题被忽略，后续又重新问了一遍。

**正确做法**：
- Phase 1 每轮只问 1-2 个问题，且明确哪个是**必须回答**的
- 用户有现成文档的情况下，澄清问题应该是"文档中没有说清楚的地方"，而不是从头走访谈
