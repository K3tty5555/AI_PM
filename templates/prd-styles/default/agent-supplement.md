# Agent 增量包 (agent-supplement.md)

> 本文件是 `feishu-template.md` 的 Agent 增量包，用于在 PRD 中描述 AI 自主决策、对话流、能力归属等传统模板覆盖不到的内容。

---

## 一、适用范围

| 产品类型 | 用法 | 增量章节 |
|---------|------|---------|
| **传统产品** | 不引入本文件 | 无 |
| **Agent 产品** | 完整引入 §A1–§A10 | 全部 10 节 |
| **混合产品** | 最小集 + 按需扩展 | 必填 §A2/§A4/§A5/§A7/§A9，推荐 §A3/§A10 |

> 判断标准：用户的核心交互是"主动操作走流程"还是"表达意图让 AI 替他做"。前者不用本包，后两者用本包。

# 第九章 Agent 专项设计

## 9.1 意图分类与触发（§A1）

### 9.1.1 触发方式

| 触发类型 | 是否支持 | 说明 |
|---------|---------|------|
| 进入页面自动激活 | ☐ | 例：打开班级学情页 → AI 助手自动展开欢迎 |
| 用户点击入口按钮 | ☐ | 例：右下角浮窗点击 |
| 文字输入触发 | ☐ | 例：用户在输入框打字 |
| 语音触发 | ☐ | 例：长按语音键 |
| 快捷指令（chip / @） | ☐ | 例：点击"找一道题"chip |
| 系统事件触发 | ☐ | 例：考试结束自动推送报告解读 |

### 9.1.2 意图清单

| 意图 ID | 用户典型说法（≥3 条） | 期望结果 | 优先级 |
|--------|---------------------|---------|-------|
| INT-01 | "找一道初一勾股定理的中等难度题" / "搜一下二次函数的高考题" / "我要勾股定理的填空题" | 返回 3-5 道符合条件的题，带题源 | P0 |
| INT-02 | ... | ... | ... |

### 9.1.3 边界与歧义处理

| 情况 | 处理策略 |
|------|---------|
| **跨意图**：一句话包含多个意图 | 拆分确认，按优先级先处理一个，提示"另一个我记下来了" |
| **模糊意图**：识别置信度 < 阈值 | AI 主动复述："我理解你想 X，对吗？"等用户确认 |
| **否定意图**：用户说"不要这个" | 撤回上一步动作，不主动建议下一步 |
| **能力外意图** | 见 §A2 Out-of-scope |

> **写作 tips**：意图清单不要写得像分类法（"知识点查询/题型查询/难度查询"），而要写**用户怎么说**。每个意图至少列 3 条真实说法，方便算法做意图分类训练集。

## 9.2 能力边界（§A2）

### 9.2.1 In-scope · 能做的

| # | 能做的事 | 对应意图 |
|---|---------|---------|
| 1 | 按知识点+难度+题型搜单道题 | INT-01 |
| 2 | 按章节范围批量出题（≤20 道） | INT-02 |
| 3 | ... | ... |

### 9.2.2 Out-of-scope · 不能做的

| 用户请求 | Agent 响应策略 | 文案示例 |
|---------|---------------|---------|
| 让 AI 评价某老师的教学水平 | 拒绝 + 引导回主场景 | "这类问题我没有依据评价，要不咱们看看具体哪个班的成绩数据？" |
| 让 AI 替学生写作业答案 | 拒绝 + 解释合规边界 | "学生作业要他们自己做才有效，我可以帮你出题或检查思路。" |
| ... | ... | ... |

### 9.2.3 Edge Cases · 模糊地带

| 场景 | 处理策略 | 理由 |
|------|---------|------|
| 用户上传一张其他平台的截图让 AI 解析 | 不解析非智学网内容，引导上传到卷库 | 数据来源不可信，无法溯源 |
| 用户连续 3 轮问同一类越界问题 | 保持一致拒绝，不升级，必要时给反馈入口 | 防止"绕过式"越狱 |
| ... | ... | ... |

> **写作 tips**：Edge Cases 是评审会上最值钱的部分，也最能体现产品思考深度。优先列出 3-5 个**看起来该做但有风险**的场景。

## 9.3 自主性等级（§A3）

### 9.3.1 等级定义（中文化）

| 等级 | 描述 | 对应交互 |
|------|------|---------|
| **L1 · 只给建议** | AI 给方案，人决定一切 | "这是 5 个候选题，你自己选" |
| **L2 · 替你做但你确认** | AI 草拟，人点确认才生效 | "我组了一份卷，预览看下，确认后入库" |
| **L3 · 默认替你做，可改可撤销** | AI 直接做，用户随时调整 | "已自动按 6:3:1 难度分布，你可以拖动修改" |
| **L4 · 异常时才打扰你** | AI 全自动，仅边界场景上报 | 后台批量出卷，结果异常才通知 |
| **L5 · 完全自动** | 不需要人 | 定时任务、纯后台 |

### 9.3.2 本期定档

> **本期目标等级：L___**
> **MVP 先做到：L___**
> **理由**：（例：教育场景容错率低，先 L2 留人确认；用户信任建立后再升 L3）

### 9.3.3 不同动作的等级映射

| 动作 | 等级 | 兜底 |
|------|------|------|
| AI 推荐题目 | L1 | 用户自己选，AI 不替选 |
| AI 组卷预览 | L2 | 必须用户点"确认入库"才落地 |
| AI 默认难度配比（6:3:1） | L3 | 用户可在确认前调整 |
| AI 自动剔除已考过的题 | L4 | 默认开启，用户在设置里关 |

> **写作 tips**：等级不是越高越好。教育、医疗、金融场景普遍 L2 起步，L3 要有强撤销机制，L4+ 必须有线上抽检。

## 9.4 AI 自主决策清单（§A4）⭐

> 这是研发评审会上最容易遗漏、最容易"扯皮"的一节。**一定要列**。
>
> 区分：
> - **工具调用**：用户说"我要 X"，AI 调 X 工具（执行者）
> - **自主决策**：用户没说细节，AI 替用户选了默认值（决策者）
>
> 自主决策必须在 PRD 里显式列出来，否则评审时研发不知道哪些是写死的、哪些是用户能改的。

### 9.4.0 决策呈现形态

> AI 替用户做的多个默认决策，怎么呈现给用户？决策呈现形态决定了交互节奏和用户信任建立方式。**这一节定形态，下一节 §9.4.1 列具体决策**。

| 形态 | 适用场景 | 节奏 | 例 |
|------|---------|------|---|
| **参数表单** | 用户清楚自己要什么、字段 ≤ 5、决策结果可预期 | 输入 → 提交 → 结果 | 不推荐用于 agent 主交互 |
| **摘要预确认（对话气泡）** | AI 替用户做了多个默认决策、用户需要快速确认/微调 | 意图 → AI 复述确认 → 生成 → 改 | V1.2 组卷 |
| **直接呈现结果 + 改入口** | 决策少、用户能从结果反推 AI 的判断 | 意图 → 直接出结果 → 用户决定 | V1.1 搜题 |
| **过程透明 + 中途介入** | 长链路任务、用户需要看到 AI 在做什么 | 意图 → 边做边播 → 用户随时打断 | 后台批量任务 |

**选择依据**：
- 决策点 ≥ 3 个 → 推荐"摘要预确认"，避免用户看到结果才发现 AI 理解错
- 决策点 ≤ 2 个 → 推荐"直接呈现结果"，少一步对话减摩擦
- 不可逆 / 高代价决策（发布、写库、扣费）→ 必须有显式确认步

### 9.4.1 决策清单

| 决策点 | AI 默认值 | 默认依据 | 用户能否修改 | 修改路径 | 是否记忆 |
|-------|----------|---------|-------------|---------|---------|
| 组卷难度分布 | 易:中:难 = 6:3:1 | 教研经验 + 学情聚合 | ✅ 能 | 卷面预览卡的拖拽条 | 修改后默认值更新 |
| 组卷题量 | 25 题 | 单元测试常规规模 | ✅ 能 | 自然语言 / 拖拽条 | 不记忆，每次默认 25 |
| 是否剔除近 30 天考过的题 | ✅ 默认开启 | 避免重复练习 | ✅ 能 | 设置项 | 班级级别记忆 |
| 出题来源 | 优先校本题库 → 区域题库 → 全网 | 数据归属优先 | ❌ 不能改 | - | - |

### 9.4.2 决策错误的回路

| 场景 | 用户感知 | 修正方式 |
|------|---------|---------|
| AI 默认配比与用户实际意图不符 | 看到预览卡发现不对 | 拖拽条调整 / 自然语言改 |
| AI 自动剔除了用户想保留的题 | 看不到这道题 | 在设置里关闭"剔除已考"开关 |
| AI 选择的题源用户不认可 | 看到题源标签后想换源 | 当前不可改，下个版本支持 |

### 9.4.3 决策权属约束

> **写作 tips**：每个决策点都要回答"这个值是产品定的还是 AI 决的"——
> - **产品定**：写死规则，不依赖模型推理（出题来源优先级）
> - **AI 决**：模型每次根据上下文动态推理（默认题量、配比）

### 9.4.4 业务规则约束（不可妥协的产品红线）

> 区别于 §9.4.1 的"默认值决策"（用户可改），本节列**产品对 AI 行为的硬约束**，研发不能自决，老师也无权关闭。

| 约束 | 为什么必须 | 不遵守的后果 |
|------|-----------|-------------|
| 例：必须先预览再发布 | 防 AI 误解直接发布学生作业 | 错题流入学生练习链路，影响学习信任 |
| 例：建组前必须查重 | 防止分组列表越建越乱 | 长期数据治理成本上升 |
| 例：失败时保留上下文，不让用户重说 | 中断后体验断崖 | 用户放弃使用 |
| 例：跨用户/跨班级数据严格隔离 | 合规底线 | 数据泄露事件 |

> **写作 tips**：业务规则约束 ≤ 5 条，多了说明没区分清楚"约束"和"默认值"。约束 = 不能让 AI/用户/研发动的；默认值 = 用户可改的。

## 9.5 工具/能力清单（§A5）

### 9.5.1 业务能力清单（业务语言）

> **不写技术接口名**（如 `GET /api/questions`），写**业务能力**：解决什么问题、给谁用、什么时候用。

| 能力名 | 解决的问题 | 触发时机 | 输入 | 输出 | 是否需审批 |
|-------|----------|---------|------|------|----------|
| 题目元数据查询 | 根据知识点+难度+题型筛题 | 用户提搜题意图 | 知识点列表、难度区间、题型 | 题目 ID + 题面摘要 + 题源 | ❌ |
| 试卷库列卷 | 找历史样卷 | 用户提"参考某次月考" | 卷子描述、时间范围 | 卷面摘要列表 | ❌ |
| 反向解析样卷 | 把样卷拆解为题目分布特征 | 用户选定一份样卷做参考 | 试卷 ID | 难度分布、知识点分布、题型分布 | ❌ |
| 卷面入库 | 把 AI 组好的卷子保存到卷库 | 用户点"确认入库" | 题目 ID 列表、卷子元信息 | 卷子 ID | ✅ 用户确认 |

### 9.5.2 上下文双层说明

**Agent 整体上下文**（贯穿所有意图）：

| 信息 | 说明 |
|------|------|
| 人设 / 系统 prompt | 见 §A6.1 |
| 用户画像 | 当前老师所属学校、年级、学科、班级 |
| 全局约束 | 数据隔离规则、合规底线 |
| 长期记忆 | 是否记住老师偏好（如组卷习惯）：**记忆/不记忆** |
| 跨会话隔离 | 不同老师 / 不同班级数据互不可见 |

**每个环节上下文**（一次具体调用时）：

| 信息 | 说明 |
|------|------|
| 当前意图 + 槽位 | 例：意图=组卷，槽位={章节: 二次函数, 题量: ?, 难度: ?} |
| 历史会话 | 保留最近 N 轮（建议 3-5 轮） |
| 上一步工具返回 | 例：题目元数据查询返回的 50 道候选题 |
| 用户上一步操作 | 例：用户拖拽过难度条到 5:4:1 |

> **写作 tips**：研发会问"上下文怎么截断、跨会话记不记、用户切场景上下文清不清"。这一节就是给他们答案的。

## 9.6 行为契约（§A6）

### 9.6.1 人设

| 维度 | 定义 |
|------|------|
| 名字/称谓 | 例：智学小助 / 不起名 |
| 语气 | 专业 / 亲切 / 简洁 / 偏正式 |
| 身份自述 | 必须表明 AI 身份（合规要求），不假装人 |
| 称呼用户 | "老师" / "您" / 直呼姓名 |

### 9.6.2 输出风格

| 维度 | 规则 |
|------|------|
| 第一人称 | 用 "我" 还是不用 |
| 表情符号 | 不用 / 限定场景用 |
| 加粗/列表/表格 | 数据类输出强制用表格 |
| 长回复组织 | 先结论后解释（B 端推荐） |
| 语言 | 仅简体中文 |
| 长度 | 单次输出 ≤ N 字 / N 段 |

### 9.6.3 拒绝与澄清话术

| 场景 | 推荐话术 | 反模式 |
|------|---------|-------|
| 超出能力 | "这超出我的能力范围，但我可以帮你 X" | "我不会" / "我只是 AI" |
| 信息不足 | 一次只问最关键的 1-2 个槽位 | 一次问 5 个 / 用户没说完就追问 |
| 可能答错 | "根据 X，我的理解是 Y，但建议你再确认 Z" | 假装很确定 / 频繁"也许""可能"稀释回答 |

### 9.6.4 Few-shot 示例（≥ 3 组）

> **写作 tips**：PM 写骨架，标 `[算法补完]` 占位，由算法在后续迭代里完善。不要硬凑。

```
示例 1：意图清晰
Input: 找 5 道初二勾股定理的中等难度题
Good Output:
  [算法补完]
  ↑ 为什么好：直接给题面摘要 + 题源 + 一键加入按钮
Bad Output:
  [算法补完]
  ↑ 为什么不好：长篇道歉、没题面、没下一步按钮
```

## 9.7 失败与兜底（§A7）⭐

> **写作 tips（重要）**：本节**只列用户在对话中真实会碰到、需要 AI 给出明确回应的场景**。技术层失败（接口超时降级、JSON Schema 校验、Token 限流等）由研发自决，不在 PRD 失败表里。
>
> 区分原则：用户能不能感知到？
> - **能感知（写）**：AI 答错、说出库里没有的数据、用户不满意、业务数据不足、越权请求
> - **不能感知（不写）**：接口超时自动重试、模型 fallback 到备用、缓存命中失败

### 9.7.1 失败模式表

| 失败类型 | 典型表现 | 检测方式 | 兜底策略 | Owner |
|---------|---------|---------|---------|-------|
| 模型答错/幻觉 | 说出题库里没有的题 | 与题库 ID 列表对照 | 强制带题源 + 不存在的 ID 直接丢弃 | 算法 |
| 工具超时 | 题库查询 > 3s | 超时阈值 | 友好降级"系统繁忙，3 秒后重试" | 后端 |
| 工具返回异常 | 字段缺失 / 格式错 | Schema 校验 | 进入错误处理分支，不展示给用户 | 后端 |
| **业务数据不足** | 题库返回结果不够 / 学情数据为空 / 对标找不到 | 计数 + 阈值校验 | AI 主动给降级方案问用户"行不行"，不直接报错 | 后端 + 产品 |
| 用户对结果不满意 | 用户说"不对" / 点"不喜欢" | 显式反馈 | 转入修改对话流（见 §A6.3） | 产品 |
| 越权请求 | 用户请求查别校数据 | 权限校验 | 拒绝 + 解释 | 后端 + 算法 |
| 对抗注入（Prompt Injection） | 用户输入"忽略以上指令" | 输入扫描 | 忽略注入 + 记录 | 算法 |

> **写作 tips**：业务数据不足是 B 端 agent 最高频的失败模式（远比模型幻觉常见）。每条业务能力都要预设"返回不够 / 返回为空"的降级文案——AI 不要直接说"出错了"，而是主动给替代方案问用户"行不行"。

### 9.7.2 Human-in-the-Loop 介入点

| 场景 | 处理 |
|------|------|
| AI 组卷涉及全年级 / 跨班级 | 必须教研组长审核后才下发 |
| AI 输出的解析涉及标准答案 | 标注"AI 初稿，请教师审核" |
| 连续 3 轮用户表达不满意 | 主动提出"要不要换我的同事帮你？"（转人工 / 转传统功能） |
| 模型置信度 < 0.6（如果可获取） | 输出 + "供参考，请人工核对" |

> **写作 tips**：HITL 不是越多越好。每个介入点要写清"什么条件触发、谁来介入、介入到什么程度"，避免变成兜底兜不动。

## 9.8 评测方案（轻量版）（§A8）

### 9.8.1 评测维度

| 维度 | 子指标 | 度量方式 | 基本可用 | 上线门槛 | 持续监控 |
|------|-------|---------|---------|---------|---------|
| 任务完成率 | End-to-end 成功率 | 评测集 + 人工标注 | 主流程跑通 | 显著高于人工基线 | ✅ |
| 幻觉率 | 编造题/答案 | 与题库 ID 对照 | 可识别 | 无 P0 误导 | ✅ |
| 拒答合理性 | 该拒不拒 / 不该拒拒了 | 双人标注 | 主要场景对 | ≥ 90% | 抽样 |
| 安全合规 | 红队对抗通过率 | 对抗集 | 关键越狱不通过 | 100% | ✅ |
| 延迟 | 端到端 P95 | 埋点 | 不卡死 | ≤ X 秒 | ✅ |

> **本阶段建议**：用"基本可用 / 上线门槛"两档**定性表述**，不硬塞百分比。等数据积累后再升级量化。

### 9.8.2 评测集来源

- **静态测试集**：N 条标注用例，来源（标注规范、版本管理）
- **动态对抗集**：红队定期更新，越狱测试用例
- **线上抽样**：N% 比例，脱敏，人工复审流程
- **A/B 基线**：对比对象（旧版 / 人工基线 / 竞品）

> **写作 tips**：评测集建设是慢工。PRD 阶段先约定**谁来建、什么节奏、第一版规模**，不要等开发完才开始想。

### 9.8.3 上线后健康度自检

> 上线初期不一定能拿到所有指标，但**关键交互的"用户接受度信号"必须埋点**。这是产品级的快速反馈回路，不依赖完整评测体系。

| 关键交互 | 接受度信号 | 阈值 | 不达标怎么办 |
|---------|----------|------|------------|
| 摘要预确认 | "对，开始组卷"按钮直接确认率 | > 80% 保留 / < 50% 简化 | 简化为跳过摘要直出 |
| 卷面预览 | 第一次预览不退回重组的比例 | > 65% 保留 | 调整默认决策值 |
| 自然语言修改 | 用户输入文字而非点按钮的比例 | > 30% 保留 | 加更多按钮 fallback |

**填写要求**：每个关键交互都要预设**两个数字**：
- **保留阈值**（>X 说明设计成立）
- **触发简化的阈值**（<Y 说明这一步是负担，要砍）

> **写作 tips**：这是 B 端 agent 试点期最聪明的健康度自检——拿不到精确评测数据时，用"用户用脚投票"的行为信号代替。

# 嵌入到既有章节的内容

## §A9 嵌入「五、全局说明」5.6：AI 入口与权限边界（混合产品必填）⭐

> 混合产品（如智学网这类传统业务 + AI 助手嵌入）必须有这一节，否则研发不知道 AI 装在哪、什么时候出现、跟原页面权限怎么协调。

### 复用前版机制

> 如果本版是已有产品的迭代（如 V1.2 是 V1.1 的迭代），可在此节直接写：
>
> > "入口位置/权限边界沿用 V1.1 PRD §5.6，本版仅以下变化：..."
>
> 但**第一版混合产品 PRD（如 V1.0 / V1.1）必须独立写完整本节**，后续版本才能复用。

### 5.6.1 入口位置

| 场景 | 入口形态 | 出现条件 |
|------|---------|---------|
| 班级学情页 | 右下角浮窗（吉祥物） | 进入页面自动出现，3 秒后自动收起 |
| 考试报告页 | 顶部 AI 摘要卡 | 报告加载完成后展示 |
| 题库页 | 搜索框右侧"问 AI"按钮 | 始终显示 |

### 5.6.2 权限边界

| 角色 | 原页面权限 | AI 助手权限 | 冲突时优先级 |
|------|----------|-----------|-------------|
| 任课教师 | 看本班数据 | 同原页面 | 取交集（更小） |
| 学情管理员 | 看全年级 | 同原页面 | 取交集 |
| 考试管理员 | 看全校 + 创建考试 | 仅查询，**不能创建/编辑** | AI 助手权限 |

### 5.6.3 数据可见性

- AI 看得到什么：对应角色权限范围内的数据
- AI 看不到什么：跨班级 / 跨年级 / 跨学校 / 学生 PII 字段（学号、家庭住址）
- 跨用户隔离：每个老师的会话独立，不共享上下文
- 是否被用于训练：**否**（合同明确）

### 5.6.4 入口失效场景

| 场景 | AI 助手处理 |
|------|-----------|
| 网络异常 | 隐藏入口，不展示假按钮 |
| AI 服务宕机 | 入口灰显 + 提示"AI 助手维护中" |
| 用户禁用 | 入口隐藏，设置项可重开 |

## §A10 嵌入「二、需求分析」2.3：为什么不用 Agent ⭐

> 反向论证。哪些场景我们**不让 AI 介入**，用传统功能。防止"AI 万能化"导致的设计膨胀。

### 2.3.1 本期不让 AI 做的事

| 场景 | 为什么不让 AI 做 | 用什么传统功能替代 |
|------|----------------|-----------------|
| 学生成绩录入 | 准确性要求高，AI 识别错误成本大 | 教师手动录入 + 批量导入 |
| 考试规则配置 | 涉及合规与权限，规则需精确 | 管理员表单配置 |
| 班级管理（增删改成员） | 操作不可逆，AI 误操作影响大 | 管理员手动操作 |
| 复杂数据图表生成 | 用户需要精确图表配置 | 仪表盘配置项 |

### 2.3.2 后续可能放开的边界

| 场景 | 触发条件 | 计划版本 |
|------|---------|---------|
| 成绩异常自动标记 | 试点学校样本 ≥ N | v1.5 |
| 考试规则自然语言配置 | 法务/合规通过 | v2.0+ |

> **写作 tips**：这一节是写给 boss 和评审委员会的。让他们看到我们**有边界感**，不是"什么都让 AI 做"。
