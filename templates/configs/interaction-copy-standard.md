# 交互文案规范

## 定位

本规范适用于 AI_PM 所有技能文件中面向用户展示的交互提示语。
目标：文案读起来像产品经理自己写的，不像机器生成的。

## 强制要求

新增或修改交互文案时，必须通过 `Humanizer-zh` 处理。

触发时机：
- 新增技能文件中的用户提示语
- 修改现有技能的交互对话文案
- 添加新的引导流程、确认弹窗、错误提示

执行方式：
1. 提取需要处理的文案片段
2. 调用 `Humanizer-zh` skill
3. 用优化后的版本替换回原文件

---

## Humanizer 处理原则

优化目标（删除 AI 写作特征）：

| 特征 | 说明 | 处理方式 |
|------|------|---------|
| 宣传性语言 | "强大的"、"智能的"、"全面的" | 删掉，直接说功能 |
| 三段式法则 | 每段都是三点列举 | 打破对称，按实际内容来 |
| AI 词汇 | "赋能"、"深度"、"精准" | 换成日常用词 |
| 过多连接词 | "首先...然后...最后" | 精简，能省则省 |
| 破折号滥用 | 每句话都有"——" | 改用逗号或分句 |
| 否定式排比 | "不只是...更是..." | 直接说是什么 |
| 模糊归因 | "业界普遍认为" | 删掉或说明具体来源 |

## Before / After 示例

### 示例 1：确认提示

❌ Before（AI味）：
```
✅ 需求澄清阶段已顺利完成！接下来将进入需求分析阶段，
系统将深度挖掘您的产品需求，为后续 PRD 生成提供强有力的支撑。
是否继续？
```

✅ After（自然）：
```
需求澄清完了，下一步是需求分析。继续？
```

### 示例 2：空状态提示

❌ Before（AI味）：
```
📚 知识库暂无内容
您可以通过以下方式开始构建您的专属产品知识图谱：
• 完成项目后使用 knowledge sync 提取经验
• 使用 knowledge add 手动添加知识
```

✅ After（自然）：
```
知识库还没有内容。
做完项目用 knowledge sync，或者直接 knowledge add 写进来。
```

### 示例 3：选择引导

❌ Before（AI味）：
```
为了更好地满足您的产品开发需求，请选择最适合您当前场景的启动方式：
1. 直接描述需求（适合思路清晰的场景）
2. interview 模式（适合有客户在场的场景）
3. data-insight（适合数据驱动决策的场景）
```

✅ After（自然）：
```
怎么开始？
  1. 直接说需求
  2. interview — 带客户现场用
  3. data-insight — 有数据文件先分析
```

### 示例 4：完成确认

❌ Before（AI味）：
```
🎉 恭喜！原型生成成功完成！
项目产出已经全部准备就绪，您可以查看以下文件：
```

✅ After（自然）：
```
原型好了。
```

### 示例 5：错误提示

❌ Before（AI味）：
```
⚠️ 系统检测到当前项目尚未完成需求澄清阶段，
无法直接进入 PRD 生成流程，请先完成前置步骤。
```

✅ After（自然）：
```
需求澄清还没做，先把那步完成再来生成 PRD。
```

## 检查清单

提交新文案前自查：
- [ ] 没有"赋能"、"强大"、"智能"等宣传词
- [ ] 没有三段式完美对称结构
- [ ] 每句话都能更短吗？
- [ ] 读出来会不会觉得像在背稿子？
