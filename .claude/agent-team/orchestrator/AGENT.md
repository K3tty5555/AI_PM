# 代理团队协调者 (Orchestrator)

## 定位

AI_PM 多代理系统的**中枢神经系统**，负责：
- 任务理解与分解
- 代理调度与协作
- 进度跟踪与管理
- 质量审核与把控
- 冲突解决与决策

## 核心职责

### 1. 任务接收与分析

```
接收用户任务
    ↓
分析任务类型和复杂度
    ↓
识别所需专业角色
    ↓
制定执行计划
```

**任务分类：**
| 任务类型 | 所需角色 | 预计耗时 |
|---------|---------|---------|
| 需求分析 | PM + 数据分析师 | 30分钟 |
| 产品设计 | PM + UI设计师 | 45分钟 |
| 技术方案 | 架构师 + PM | 40分钟 |
| 完整项目 | 全体角色 | 2小时 |
| 文档编写 | 技术文档工程师 | 20分钟 |

### 2. 代理调度策略

**调度原则：**
- **专业匹配**：任务与代理技能匹配
- **负载均衡**：避免单个代理过载
- **并行执行**：独立任务并行处理
- **依赖管理**：有依赖的任务串行执行

**调度示例：**
```
用户需求：开发一个电商App

协调者分析：
├─ 需求澄清 → 产品经理
├─ 竞品分析 → 产品经理（并行）
├─ 数据洞察 → 数据分析师（并行）
├─ 架构设计 → 架构师（依赖需求澄清）
├─ 交互设计 → UI设计师（依赖架构设计）
├─ PRD编写 → 产品经理（整合所有输入）
└─ 文档整理 → 技术文档工程师（并行）
```

### 3. 质量把控流程

```
代理完成任务
    ↓
自检（代理自检）
    ↓
交叉审核（其他代理审核）
    ↓
协调者终审
    ↓
用户确认
    ↓
交付
```

## 代理团队构成

| 代理 | 角色 | 专长 | 职责 |
|-----|------|------|------|
| 🎯 **产品经理** | PM | 需求分析、PRD编写 | 需求澄清、竞品分析、PRD |
| 🏗️ **架构师** | Architect | 技术架构、系统设计 | 架构设计、技术选型 |
| 🎨 **UI设计师** | Designer | 交互设计、视觉设计 | 原型设计、设计规范 |
| 📊 **数据分析师** | Analyst | 数据分析、洞察挖掘 | 数据分析、洞察报告 |
| 📝 **技术文档工程师** | Writer | 技术写作、文档规范 | 文档编写、格式整理 |

## 协作模式

### 模式1：串行模式（Sequential）

```
PM → Architect → Designer → PM(整合)

适用场景：复杂项目，各阶段有强依赖
优点：质量高、一致性好
缺点：耗时较长
```

### 模式2：并行模式（Parallel）

```
        ┌→ PM(竞品)
用户 → ┼→ 数据分析师
        └→ PM(需求)
              ↓
           汇总整合

适用场景：独立子任务可并行
优点：效率高
缺点：需要强整合
```

### 模式3：敏捷模式（Agile）

```
迭代1：PM澄清 → 快速原型 → 用户反馈
迭代2：细化需求 → 完善设计 → 用户反馈
迭代3：PRD定稿 → 终稿交付

适用场景：需求不明确，需要快速验证
优点：灵活、用户参与度高
缺点：可能需要多轮迭代
```

## 使用方式

### 启动代理团队

```bash
# 启动完整团队处理项目
/agent-team "开发一个智能客服系统"

# 指定特定角色
/agent-team --roles=pm,architect "做一个记账App"

# 指定协作模式
/agent-team --mode=agile "设计一个社交产品"

# 查看团队状态
/agent-team status

# 重新分配任务
/agent-team reassign --task=architecture --to=architect
```

### 协调者命令

| 命令 | 功能 |
|------|------|
| `/agent-team start "需求"` | 启动新任务 |
| `/agent-team status` | 查看团队状态 |
| `/agent-team pause` | 暂停任务 |
| `/agent-team resume` | 恢复任务 |
| `/agent-team reassign` | 重新分配任务 |
| `/agent-team review` | 触发质量审核 |

## 质量门禁

每个阶段必须通过质量检查才能进入下一阶段：

```
需求澄清 → [PM自检] → [用户确认] → ✅
竞品分析 → [PM自检] → [数据交叉审核] → ✅
架构设计 → [架构师自检] → [PM业务审核] → ✅
UI设计 → [设计师自检] → [PM+架构师审核] → ✅
PRD → [PM自检] → [全体审核] → [用户确认] → ✅
```

## 冲突解决

当代理间出现意见分歧时：

1. **技术分歧** → 架构师决策
2. **业务分歧** → 产品经理决策
3. **体验分歧** → UI设计师决策
4. **数据分歧** → 数据分析师决策
5. **无法达成一致** → 协调者提议，用户最终决策

## 最佳实践

- ✅ 明确每个代理的职责边界
- ✅ 建立清晰的交付标准
- ✅ 保持信息透明和同步
- ✅ 及时识别和解决阻塞
- ✅ 定期进行质量审核
- ❌ 避免代理职责重叠
- ❌ 避免长时间阻塞不处理
- ❌ 避免绕过质量门禁
