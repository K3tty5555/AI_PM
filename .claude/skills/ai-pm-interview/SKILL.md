# 现场调研/客户访谈模式

## 概述

现场调研模式专为**线下与客户面对面沟通**场景设计，帮助你：

- 🎤 **结构化访谈** - 按框架提问，不遗漏关键点
- 📝 **实时记录** - 记录客户原话，保留真实需求
- ⚡ **现场生成** - 快速产出 PRD/原型，当场展示
- 🔄 **迭代调优** - 根据反馈快速调整，反复确认

## 适用场景

| 场景 | 示例 |
|-----|------|
| 内部业务方调研 | 去销售部门调研 CRM 改进需求 |
| 外部客户访谈 | 拜访大客户，收集产品反馈 |
| 现场需求确认 | 在客户现场确认需求细节 |
| 敏捷迭代沟通 | 与核心用户快速迭代方案 |

## 工作流程

```
Phase 0: 准备阶段（访谈前）
    ↓ 了解已有功能背景
    ↓ 准备访谈提纲

Phase 1: 现场访谈（与客户面对面）
    ↓ 结构化提问
    ↓ 实时记录客户反馈

Phase 2: 快速生成（现场产出）
    ↓ 基于访谈记录生成 PRD/原型
    ↓ 现场展示给客户

Phase 3: 迭代调优（反复沟通）
    ↓ 收集客户反馈
    ↓ 快速调整方案

Phase 4: 导出交付
    ↓ 生成完整文档
```

```markdown
## 结构化问题清单

### 【开场问题】建立语境
1. 请介绍一下您目前如何使用这个功能？
   - 使用场景是什么？
   - 使用频率如何？
   - 一般在什么情况下使用？

2. 这个功能对您的工作有什么价值？
   - 解决了什么问题？
   - 带来了什么便利？

### 【痛点深挖】发现问题
3. 在使用过程中，最让您困扰的是什么？
   - 具体是什么问题？
   - 这个问题出现的频率是？（每天/每周/偶尔）
   - 这个问题对您的工作有什么影响？

4. 您现在是怎么解决这个问题的？
   - 有什么 workaround 吗？
   - 为此付出了什么额外成本？

5. 还有其他让您不满意的地方吗？

### 【需求澄清】明确期望
6. 您希望理想情况下这个功能是什么样的？
   - 如果能魔法般改变，您希望怎样？
   - 有没有参考的产品/功能？

7. 如果只能改一个地方，您最希望改什么？
   - 为什么是这个？

9. 您提到的这几个需求，按重要程度排序是？
   - 必须有 vs 最好有 vs 可以有

10. 如果资源有限，哪些是可以暂缓的？

### 【验收标准】确认成功标准
11. 怎么算这个功能做好了？
    - 您会如何验证？
    - 成功的标准是什么？

## 访谈技巧提醒
- ✅ 多倾听，让客户多说（少说多听）
- ✅ 追问具体场景（"能举个例子吗？"）
- ✅ 确认理解（"我理解您说的是...对吗？"）

- ✅ 关注情绪变化（叹气、皱眉、兴奋）
```

## Phase 2: 快速生成

### 生成确认

访谈结束（用户输入 "完成"）：

```
✅ 访谈完成！

📊 需求摘要：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【核心痛点】（按严重程度）
1. 🔴 手工整理报表耗时（30分钟/天）
2. 🟡 容易出错，需反复核对
3. 🟢 导出格式不满足要求

【客户诉求】
• 自动化报表生成
• 一键导出标准格式
• 支持按区域/时间筛选

【优先级】
P0（必须有）：自动报表生成
P1（最好有）：一键导出
P2（可以有）：多种格式支持

【验收标准】
• 报表生成时间 < 1分钟
• 无需人工整理
• 格式符合领导要求

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

💬 下一步：
1. "生成PRD" → 生成详细PRD文档
2. "生成原型" → 生成可交互原型
3. "先生成草图" → 生成低保真草图（推荐，更快）
4. "返回修改" → 返回访谈补充
```

### 快速生成策略

**现场生成原则**：
1. **速度优先** - 先生成低保真草图，确认方向
2. **客户易懂** - 避免技术术语，多用图示
3. **重点突出** - 只展示核心功能和改动点
4. **快速迭代** - 生成 → 展示 → 反馈 → 调整

**生成内容**：

1. **PRD 草案**（简化版）：
   - 只包含核心功能描述
   - 用表格展示改动前后对比
   - 配简单流程图

2. **低保真草图**：
   - 手绘风格线框图
   - 标注关键改动点
   - 可用 ASCII 或简单 HTML

3. **高保真原型**（可选）：
   - 在草图确认后生成
   - 可交互的 HTML 原型

### 生成输出

创建文件：
- `11-field-research/03-prd-draft-v1.md` - PRD 草案
- `11-field-research/04-prototype-v1/` - 原型（草图或高保真）

## 与 Agent Team 的协作

现场调研模式可与 **Agent Team** 深度协作，利用专属角色提升调研效率：

### 专用角色

| 角色 | 职责 | 协作阶段 |
|-----|------|---------|
| 🎤 **用户研究员** | 执行结构化访谈、实时记录、洞察分析 | Phase 1 |
| 🧩 **需求整合师** | 提取需求、整合信息、优先级排序 | Phase 2 |

### 使用方式

```bash
# 启动现场调研专用团队
/agent-team --mode=serial \
  --roles=user-researcher,requirement-synthesizer,pm,designer \
  "后台管理系统-订单模块现场调研"

# 多客户并行调研
/agent-team --mode=parallel \
  --roles=user-researcher,user-researcher,requirement-synthesizer,pm \
  "CRM系统用户体验调研"
```

### 协作流程

```
用户研究员
    ↓ 执行访谈
    输出：访谈记录、用户画像、洞察报告

产品经理
    ↓ 完善PRD

UI设计师
    ↓ 设计原型
```

## 与标准模式的关系

**现场调研模式** vs **标准模式**：

| 对比项 | 现场调研模式 | 标准模式 |
|-------|-------------|---------|
| **适用场景** | 线下现场、面对面 | 线上、异步 |
| **速度要求** | 极快（现场） | 正常 |
| **文档详细度** | 精简版 | 完整版 |
| **迭代方式** | 实时迭代 | 阶段性迭代 |
| **输出** | PRD草案+原型 | 完整PRD+原型+数据设计 |

**模式转换**：
- 现场调研结束后，可选择转入标准模式继续完善
- 标准模式可以复用现场调研的访谈记录

---

## Anti-Pattern

- ❌ 不要在现场花太长时间写详细文档
- ❌ 不要一次性问完所有问题再记录
- ❌ 不要过度解读客户需求
- ❌ 不要承诺无法实现的交付时间
- ❌ 不要让客户等待太久（生成时间要短）
