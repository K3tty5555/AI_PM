# 数据分析与埋点设计

> ⚠️ **已合并**: 本技能已合并到 `ai-pm-data`，请使用 `/ai-pm data metrics`

## 定位

AI_PM 的**数据能力中心**，确保产品从设计之初就有完整的数据支撑体系。

## 迁移提示

```
旧命令: /ai-pm analytics
新命令: /ai-pm data metrics

过渡期: 2026-03-01 ~ 2026-03-15 (2周)
```

- **指标体系设计**: 从业务目标拆解到可量化的数据指标
- **埋点方案设计**: 详细的事件定义、参数设计、触发时机
- **数据看板规划**: 可视化报表需求，支撑日常数据监控
- **实验设计**: A/B测试方案、假设验证、样本量计算

## 核心能力

### 1. 指标体系设计 (Metrics Design)

**北极星指标识别:**
```
业务目标: 提升用户留存
    ↓
北极星指标: 7日留存率
    ↓
一级指标:
  - 新用户次日留存率
  - 活跃用户周留存率
  - 功能使用留存率
    ↓
二级指标:
  - 首次使用完成率
  - 关键功能渗透率
  - 用户活跃度分布
```

**指标体系输出:**
```markdown
## 指标体系

### 北极星指标
| 指标名称 | 定义 | 计算公式 | 目标值 | 监测频率 |
|---------|------|---------|-------|---------|
| 7日留存率 | 新用户注册后7天内再次访问的比例 | 7日内回访用户数/新增用户数 | 40% | 每日 |

### 一级指标
| 指标名称 | 定义 | 当前值 | 目标值 | 负责团队 |
|---------|------|-------|-------|---------|
| 次日留存率 | ... | - | 60% | 产品组 |
| 周活跃用户 | ... | - | 10000 | 运营组 |

### 二级指标 (驱动指标)
| 指标名称 | 与北极星关系 | 监测目的 |
|---------|-------------|---------|
| 首次使用完成率 | 正向驱动 | 识别新用户流失点 |
| 核心功能渗透率 | 正向驱动 | 发现功能推广机会 |
```

### 2. 埋点方案设计 (Tracking Design)

**事件设计原则:**
- **完整性**: 覆盖用户全生命周期
- **一致性**: 命名规范统一
- **可扩展**: 预留自定义参数
- **可追溯**: 支持用户行为链路还原

**埋点事件分类:**

```markdown
### 页面浏览事件 (PageView)

| 事件名称 | 触发时机 | 页面参数 | 业务参数 |
|---------|---------|---------|---------|
| page_home_view | 首页加载完成 | page_id, referer | user_type, entry_source |
| page_detail_view | 详情页加载 | page_id, item_id | item_category, item_source |

### 点击事件 (Click)

| 事件名称 | 触发时机 | 元素标识 | 业务参数 |
|---------|---------|---------|---------|
| click_submit | 点击提交按钮 | element_id, element_text | form_type, submit_result |
| click_banner | 点击Banner | element_id, banner_id | banner_position, campaign_id |

### 功能使用事件 (Feature)

| 事件名称 | 触发时机 | 功能参数 | 结果参数 |
|---------|---------|---------|---------|
| feature_search | 执行搜索 | search_type, keyword | result_count, has_result |
| feature_share | 分享操作 | share_channel, content_type | share_result |

### 业务转化事件 (Conversion)

| 事件名称 | 触发时机 | 转化参数 | 金额参数 |
|---------|---------|---------|---------|
| conversion_register | 完成注册 | reg_source, reg_type | - |
| conversion_purchase | 完成购买 | item_id, quantity | price, currency |
```

**详细埋点文档结构:**

```markdown
## 埋点需求文档

### 事件 1: 首页浏览

**基础信息**
- 事件ID: pv_homepage
- 事件名称: 首页浏览
- 事件类型: 页面浏览
- 触发时机: 首页完全加载后
- 优先级: P0

**上报参数**
| 参数名 | 类型 | 是否必填 | 示例值 | 说明 |
|-------|------|---------|-------|------|
| page_name | string | 是 | "home" | 页面标识 |
| referer | string | 否 | "search" | 来源页面 |
| entry_source | string | 否 | "push" | 入口来源 |
| user_type | string | 是 | "new"/"active" | 用户类型 |

**技术备注**
- 上报方式: 页面加载完成后延迟100ms上报
- 去重规则: 同一会话5分钟内不重复上报
- 特殊情况: 从后台切换回前台不上报

**验收标准**
- [ ] 首页加载后事件正常上报
- [ ] 参数值符合预期
- [ ] 网络异常时有重试机制
```

### 4. A/B测试设计 (Experiment Design)

**实验设计流程:**

```markdown
## A/B测试方案

### 实验 1: 注册按钮文案优化

**实验背景**
当前注册按钮文案为"立即注册"，希望通过优化文案提升注册转化率。

**假设**
将文案改为"免费试用"，注册转化率将提升15%以上。

**实验设计**
| 项目 | 内容 |
|------|------|
| 实验变量 | 注册按钮文案 |
| 对照组(A) | "立即注册" |
| 实验组(B) | "免费试用" |
| 分配比例 | 50% : 50% |
| 目标指标 | 注册完成率 |
| 守护指标 | 次日留存率（不低于对照组） |

**样本量计算**
- 基线转化率: 25%
- 期望提升: 15%相对提升（绝对值+3.75%）
- 显著性水平: 95%
- 统计功效: 80%
- **所需样本**: 每组 2,850 人

**实验周期**
- 预计日流量: 1,000人/组
- 实验天数: 3天（达到样本量后观察2天）
- 总周期: 5天

**埋点需求**
- exp_button_register_view: 查看注册按钮
- exp_button_register_click: 点击注册按钮
- exp_register_complete: 完成注册

**所有事件需带参数:**
- experiment_id: "exp_001_button_copy"
- variant: "A" 或 "B"
```

## 最佳实践

### 埋点设计原则

1. **先业务后技术**: 先明确业务问题，再设计埋点
2. **事件原子化**: 每个事件只记录一个行为
3. **参数标准化**: 统一参数命名规范
4. **预留扩展性**: 保留自定义参数字段

### 指标设计原则

1. **少即是多**: 北极星指标只有1个
2. **可行动**: 每个指标都要有明确的行动方
3. **可对比**: 支持时间对比、分组对比
4. **防作弊**: 设计反刷量机制

### 常见反模式

❌ **埋点过多**: 事件爆炸，数据难以分析
❌ **参数缺失**: 关键维度未采集，无法下钻
❌ **口径不一**: 同一指标在不同地方定义不同
❌ **事后补埋**: 产品上线后才想起要埋点

✅ **推荐做法**: 数据设计前置，与PRD同步产出

```markdown
- 指标体系完整性
- 埋点方案可执行性
- 数据质量保障
- 分析维度充分性

**评审重点**:
- 北极星指标是否明确
- 埋点事件覆盖完整度
- 参数设计是否支持分析需求
- 是否有数据校验机制

**典型问题**:
- 缺少核心转化事件定义
- 指标口径不清晰
- 埋点参数无法支持细分分析
- 无数据质量监控方案
```
