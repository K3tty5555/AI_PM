# pm-gap-research — 知识缺口驱动的讨论提纲

## 核心原则

**从缺口出发，而非从问题出发。** 先明确这场会议要填补哪些 PRD 盲区或 Skill 参数空白，再倒推要问什么。输出是**问题列表 + 记录空白**，不是逐字稿。

---

## 执行步骤

**Step 1 · 确认缺口（先做这步）**

问用户：这次调研之后，哪些 PRD 章节或 Skill 参数能写完？

- PRD 盲区：通常是流程类（用户怎么做）、场景类（什么条件下做）、失败类（什么情况做不成）
- Skill 参数：检索权重、意图边界、隐性判断标准、用户真实表达方式

**Step 2 · 确认会议形式**

| 形式 | 对应文档风格 |
|------|------|
| 1:1 线下 / 线上 | 可追问，问题可多可细 |
| 群体线上 | 问题 ≤ 6 个，每题让 2-3 人回答即可，不追问链 |
| 大型评审 / 汇报 | 不适合用此 Skill，换 multi-perspective-review |

**Step 3 · 设计问题**

两类问题分别对应两种产出：

| 类型 | 问法特征 | 产出 |
|------|------|------|
| **场景还原类** | "最近一次…" "从 A 到 B 中间经过哪些步骤" | 填 PRD 流程 / 场景章节 |
| **行为描述类** | "你一般怎么说…" "什么情况下会觉得不合适" | 校准 Skill 权重和边界 |

主问题总数控制在 **5-6 个**，结尾必须留一题：「最头疼的一个环节是什么」（快速收口）。

**Step 4 · 输出文档**

格式固定：

```
## 模块名

- 问题1
- 问题2

记录：
（留空）
```

每个模块一个记录框，不写引导语，不写追问链。

---

## 常见错误（从 RED 阶段观察）

| 错误 | 为什么发生 | 怎么避免 |
|------|------|------|
| 写成逐字稿 | 默认「访谈 = 脚本」 | 先问会议形式，群体线上直接排除脚本 |
| 没区分 PRD 类 vs Skill 类问题 | 忽略双目的 | Step 1 强制确认两类缺口 |
| 问题太多太细 | 想「覆盖全面」 | 超过 6 个主问题时主动删减 |
| 没留记录空白 | 觉得会后再整理 | 模板固定带记录框，不可省略 |
