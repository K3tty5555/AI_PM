# 用户交互模式

## 现场调研/客户访谈模式

针对**线下与客户面对面沟通**场景的特殊工作模式，详见 [ai-pm-interview 技能](../ai-pm-interview/SKILL.md)。

### 适用场景

- 去客户现场调研已有功能改进需求
- 与业务方面对面沟通确认需求细节
- 快速从简单需求生成方案并现场展示
- 根据客户反馈实时迭代调优

### 与标准模式的区别

| 维度 | 现场调研模式 | 标准模式 |
|-----|-------------|---------|
| **场景** | 线下、面对面、实时 | 线上、异步 |
| **速度** | 极快，现场生成 | 正常，深度分析 |
| **流程** | 访谈→生成→迭代 | 分析→PRD→原型→评审 |
| **文档** | 精简版 PRD | 完整版 PRD |
| **迭代** | 实时迭代 | 阶段性迭代 |

### 快速入口

```bash
/ai-pm interview                    # 进入现场调研模式
/ai-pm interview {功能名}            # 针对指定功能
/ai-pm field                        # 简写形式
```

### 工作流程

```
Phase 0: 准备阶段
   ↓ 了解已有功能背景
   ↓ 准备访谈提纲

Phase 1: 现场访谈
   ↓ 结构化提问
   ↓ 实时记录客户反馈

Phase 2: 快速生成
   ↓ 基于访谈记录生成 PRD/原型
   ↓ 现场展示给客户

Phase 3: 迭代调优
   ↓ 收集客户反馈
   ↓ 快速调整方案

Phase 4: 导出交付
   ↓ 生成完整文档
```

### 文件结构

```
output/projects/{项目名}/
└── 11-field-research/              # 现场调研目录
    ├── 00-prep-notes.md            # 准备笔记
    ├── 01-interview-guide.md       # 访谈提纲
    ├── 02-interview-notes.md       # 访谈记录
    ├── 03-prd-draft-v1.md          # PRD草稿
    ├── 04-prototype/               # 原型
    ├── 05-iteration-log.md         # 迭代记录
    └── output/                     # 导出交付
```

## 新项目引导（三种启动方式）

当用户表示要「开始新项目」或类似意图时，主动展示三种启动方式，**特别强调「用户访谈」选项**：

### 引导话术

```
开始新项目，三种方式：

  1. 直接说需求 → 例：做一个帮用户决定吃什么的 App
  2. interview  → 有客户在场时用，边聊边出 PRD（推荐）
  3. data-insight {文件} → 从数据里找需求，支持 CSV / Excel / JSON

说需求，或输入 2 / 3 选方式。
```

### 为什么强调「用户访谈」

很多用户不知道我们有这个功能，但它非常强大：
- **面对面访谈客户**时，打开 interview 模式
- **结构化提问** + **实时记录** + **现场生成 PRD**
- 客户说完需求，当场就能看原型

### 用户选择后的处理

| 用户输入 | 处理方式 |
|---------|---------|
| 直接描述需求 | 进入「探索模式」→ 生成概念草图 |
| 输入 "interview" / "访谈" | 调用 ai-pm-interview 技能 |
| 输入 "data" / 文件路径 | 调用 ai-pm-data 技能 |

## 探索模式（概念草图）

针对用户的初步需求，在进入详细访谈前先快速生成概念草图，让用户确认方向：

### 适用场景

- 用户只输入了一句话需求（如"做一个todo app"）
- 需求比较模糊，需要先对齐大方向
- 用户希望快速看到方向是否正确

### 交互流程

**步骤1：快速理解**
```
先确认一下方向：
```

**步骤2：生成概念草图**
```
── 方向确认 ──

产品定位：{一句话定位}
目标用户：{目标用户群体}
核心功能：
   • {核心功能1}
   • {核心功能2}
   • {核心功能3}

方向对吗？
  继续深入 → 详细需求分析
  调整方向 → 告诉我哪里要改
  直接出PRD → 就按这个来
  加急 → 一路跑到原型
```

### 概念草图生成原则

- 基于用户输入提取关键信息
- 不追问，只呈现AI的理解
- 给用户选择权，让用户掌控方向
- 提供快速通道（直接PRD/加急模式）给有经验的用户

## 交互式需求澄清

如果需求信息不完整，或用户选择"继续深入"，启动深度访谈模式：

### 开场白（v2.0优化版）

```
当前项目：{项目名}

需求记下来了："{用户输入}"

有几个问题想先确认：
```

### 首要确认：项目类型（v2.0新增）

**第一个问题（最重要）：**
```
💬 请先确认项目类型：

1. 【0-1全新开发】从零开始做一个新产品/新功能
   → 我会进行完整的需求分析、竞品研究、用户画像

2. 【基于已有项目迭代】在现有产品基础上优化改进
   → 我会重点分析现有问题和改进点，关注兼容和过渡

3. 【参考竞品/对标】参考市场上的同类产品
   → 我会深入分析竞品，找出差异化和突破点

请回复：1 / 2 / 3
```

### 参考资源收集（v2.0新增）

**第二个问题（启动Phase 0参考分析）：**
```
有参考资料吗？

  现网 URL / 账号密码（迭代项目）
  竞品网址或截图
  UI 设计稿、品牌规范

直接发链接、图片或文件路径，没有就说"没有"。
```

**处理用户提供的参考资源：**
- 如果用户提供了URL → 使用 Playwright MCP 抓取网页内容
- 如果用户提供了图片 → 分析图片内容（UI/UX/竞品界面）
- 如果用户提供了账号密码 → 登录后抓取内部系统
- 如果用户提供了文件路径 → 读取文件内容

**自动触发Phase 0参考分析：**
```
📁 检测到参考资源：
   • 现网URL: https://xxx.com (账号: xxx)
   • 竞品截图: 3张
   • 设计稿: Figma链接

🔍 是否开始参考分析？
   • "开始分析" → 使用 Playwright MCP 抓取网页，分析图片
   • "跳过" → 直接进入需求澄清
```

### 访谈问题清单（按需追问）

**基础问题：**

1. **项目背景**（必问）
   - 【0-1】目标用户是谁？他们有什么痛点？
   - 【迭代】现有系统最大的问题是什么？用户反馈如何？
   - 【对标】竞品哪里做得好？我们要怎么超越？

2. **目标用户**（必问）
   - 这个产品是给谁用的？
   - 他们大概是什么年龄段？什么职业？
   - 他们在什么场景下会用这个产品？

3. **核心痛点**（必问）
   - 他们现在怎么解决这个问题？
   - 现在的解决方案有什么不好的地方？
   - 如果不做这个产品，他们会怎么样？

4. **成功标准**（必问）
   - 怎么算这个产品成功了？
   - 你期望用户达到什么样的使用频率？
   - 有没有数字化的目标？（比如留存率、付费率）

5. **竞品情况**（选问）
   - 你知道有哪些类似的产品吗？
   - 你觉得他们做得怎么样？
   - 我们要跟他们的区别是什么？

6. **约束条件**（选问）
   - 大概什么时候需要上线？
   - 技术团队规模怎么样？
   - 有没有什么必须遵守的规则或限制？

7. **商业目标**（选问）
   - 这个产品怎么赚钱？（如果有计划的话）
   - 打算免费还是付费？

**追问原则：**
- 根据用户的回答，灵活决定追问深度
- 如果用户回答详细，可以跳过后续问题
- 如果用户回答模糊，继续追问具体细节
- 每次最多问 2-3 个问题，避免压迫感

**确认保存：**
访谈结束后，生成需求草稿文件，向用户确认：
```
整理了一下：

{需求摘要}

没问题的话我就开始分析了，要改就说。
```

## 设计规范管理（独立能力）

设计规范管理由**独立技能 `ai-pm-config`** 负责。本技能通过调用该独立技能来获取设计规范列表。

### 职责分离

| 技能 | 职责 |
|------|------|
| `ai-pm-config` | 设计规范的 CRUD 管理、解析、存储 |
| `ai-pm` (本技能) | 在 PRD 生成阶段调用设计规范列表，保存用户选择到项目配置 |
| `ai-pm-prototype` | 读取项目配置，调用设计规范导出接口获取 tokens |

### 调用接口

**获取设计规范列表（Phase 5）：**
```javascript
// 调用 ai-pm-config 技能
const systems = exec('ai-pm-config list --json');
// 返回格式：[{name, description, fileCount, updatedAt}, ...]
```

**保存选择到项目配置：**
```javascript
// 保存到项目目录下的 .ai-pm-config.json
saveToProjectConfig({
  designSystem: selectedSystemName,  // 如 "my-company"
  designSystemVersion: "1.0"
});
```

### PRD 生成时选择设计规范（Phase 5）

在 PRD 生成前的统一确认节点，调用独立技能获取规范列表：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎯 关键确认：即将生成最终PRD
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📋 前序阶段核心结论：
1. 目标用户：{一句话概括}
2. 核心痛点：{Top 3 痛点}
3. 主要功能：{关键功能清单}

🎨 设计规范选择：

📁 调用 ai-pm-config list...

【情况A：检测到设计规范】
可用设计规范：
   1. enterprise-standard（企业级B端规范）
   2. apple-clean（Apple 风格规范）
   3. my-company（自定义规范）

【情况B：未检测到设计规范】
ℹ️ 未检测到设计规范，将使用默认风格
💡 提示：使用 `/ai-pm config upload {规范名}` 可上传设计规范

💬 请选择：
   • 回复数字 1/2/3 → 使用该规范（保存到项目配置）
   • "不使用规范" → 采用默认风格
   • "管理规范" → 调用 ai-pm-config 进入管理界面
```

### 与 ai-pm-config 的协作流程

```
用户执行 /ai-pm config upload my-company
    ↓
ai-pm-config 技能处理（完全独立）
    • 创建目录 templates/ui-specs/my-company/
    • 解析设计资源
    • 生成 design-tokens.json
    ↓
后续 PRD 生成时（本技能）：
    • 调用 ai-pm-config list 获取列表
    • 用户选择后保存到项目配置
    ↓
原型生成时（ai-pm-prototype 技能）：
    • 读取项目配置获取 designSystem
    • 调用 ai-pm-config export {name} 获取 tokens
    • 应用 tokens 生成 CSS
```

### 独立技能调用示例

**管理设计规范（完全委托）：**
```
用户: /ai-pm config upload my-company

AI: [调用 ai-pm-config 技能处理]
    该操作由独立技能 ai-pm-config 处理

📁 ai-pm-config 输出：
✅ 设计规范 "my-company" 创建完成！
   位置：templates/ui-specs/my-company/
   文件：README.md, design-tokens.json
```

**PRD 阶段使用：**
```
AI: 🎨 正在获取设计规范列表...
   $ ai-pm-config list --json

   返回结果：
   [1] enterprise-standard
   [2] my-company

💬 请选择要应用的设计规范（回复数字）：

用户: 2

AI: ✅ 已选择 "my-company" 设计规范
   已保存到项目配置：.ai-pm-config.json

   该配置将在原型生成阶段被 ai-pm-prototype 读取使用
```

## 产品经理风格管理（独立能力）

产品经理风格管理是一项**独立能力**，用于分析、保存和管理用户的 PRD 写作风格，供后续生成文档时选用。

### 使用场景

- 用户上传自己的 PRD 文档，AI 分析其写作风格
- 保存团队/公司的 PRD 规范作为标准风格
- 管理多种风格（B端/C端/数据产品等不同风格）
- 生成符合个人/团队习惯的 PRD 文档

### 风格库存储位置

```
templates/prd-styles/
├── default/                   # 系统默认风格
│   └── style-config.json
├── enterprise-standard/       # 企业标准风格示例
│   ├── style-config.json
│   └── examples/
│       └── sample-prd.md
├── consumer-app/              # C端应用风格示例
│   ├── style-config.json
│   └── examples/
│       └── sample-prd.md
└── [用户自定义风格]/          # 用户上传的风格
    ├── style-config.json
    └── examples/
        └── original-prd.md
```

### 命令

| 命令 | 作用 | 示例 |
|------|------|------|
| `/ai-pm config` | 进入风格管理菜单 | - |
| `/ai-pm config list` | 列出所有风格 | - |
| `/ai-pm config analyze {文件路径}` | 分析 PRD 创建风格 | `/ai-pm config analyze ~/my-prd.md` |

### 工作流程

```
用户执行 /ai-pm config
    ↓
显示风格管理菜单
    ├─ 1. 分析 PRD 创建新风格
    ├─ 2. 查看/管理现有风格
    └─ 3. 删除风格

选择 1: 分析 PRD 创建新风格
    ↓
💬 请输入风格名称（如：my-style）
    ↓
💬 请提供 PRD 文件路径
    ↓
📁 创建文件夹 templates/prd-styles/{风格名}/
    ↓
🔄 AI 分析 PRD 文档
    • 提取文档结构偏好
    • 识别写作风格特征
    • 分析格式偏好
    • 统计内容侧重
    ↓
📝 生成风格配置文件
    • style-config.json
    • 保存原始 PRD 到 examples/
    ↓
✅ 风格创建完成
```

### 风格 vs 设计规范的区别

| 对比项 | 产品经理风格 | 设计规范 |
|--------|-------------|---------|
| **关注点** | PRD 写作风格、结构、术语 | UI 视觉风格、组件、交互 |
| **存储位置** | `templates/prd-styles/` | `templates/ui-specs/` |
| **影响内容** | 文字描述、章节结构、表格格式 | 原型视觉、颜色、字体、间距 |
| **使用时机** | 生成 PRD 文档时 | 生成原型时 |
| **配置文件** | `style-config.json` | `design-tokens.json` |

**分析 PRD 创建风格：**

```

    [1/4] 分析文档结构... ✅
    [2/4] 识别写作风格... ✅
    [3/4] 提取格式偏好... ✅
    [4/4] 统计内容侧重... ✅

📊 风格特征摘要：

【文档结构】

   • 表格密度：高（平均每章2.3个表格）

【写作风格】
   • 语言风格：技术严谨型
   • 详细程度：详尽
   • 术语习惯：互联网产品术语

【格式偏好】
   • 表格字段：标准字段+自定义扩展
   • 标注方式：加粗重点
   • 优先级标记：P0/P1/P2

📝 生成风格配置：
    • templates/prd-styles/my-company-style/style-config.json
    • templates/prd-styles/my-company-style/examples/original-prd.md

✅ 风格 "my-company-style" 创建完成！

💡 后续使用：
   生成PRD时，该风格将出现在选择列表中
```
