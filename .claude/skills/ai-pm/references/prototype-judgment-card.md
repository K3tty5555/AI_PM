# Prototype Judgment Card

> 原型质量判断卡。用于 Phase 7 原型生成前的蓝图、生成中的约束、生成后的质量审计。

## 1. 原型目标

HTML 原型的标准不是"有页面"，而是"可评审"：
- 评审人能在 30 秒内看懂产品形态和主流程。
- 核心任务能被点击、切换、输入或确认走通。
- 页面有真实业务密度，不是营销模板或灰白卡片堆叠。
- 关键状态有呈现，尤其是失败、空结果、加载、成功反馈。

## 2. 生成前蓝图

生成 HTML 前必须先在内部完成以下判断：

| 项 | 必答问题 |
|----|----------|
| 页面/视图 | PRD 中哪些内容需要界面承载？哪些是无界面逻辑？ |
| 主流程 | 用户从哪里开始，经过哪些步骤完成核心任务？ |
| 信息层级 | 首屏最重要的 3 个信息是什么？ |
| 状态 | 默认、空、加载、错误、成功分别怎么呈现？ |
| 交互 | 哪些按钮/输入/切换必须有反馈？ |
| 视觉方向 | 产品场景要求高密度、轻引导、沉浸式还是工具型？ |

蓝图可以不单独展示给最终用户，但必须影响 HTML 结构和样式。

## 3. 视觉设计红线

以下问题任一出现，视觉设计不得分：

- 只有白底、灰卡片、蓝按钮，缺少场景化设计主张。
- B 端管理台做成大标题、大留白、营销落地页。
- C 端/移动端做成密集后台表格。
- 页面元素大小、间距、层级没有规律，像模型随手拼贴。
- 假数据明显敷衍：测试数据、示例内容、张三李四、无业务字段。
- 重要状态只用文字说明，没有可视化反馈区。
- 使用大量装饰性渐变、光效、卡片阴影掩盖信息不足。

## 4. 场景化视觉判断

| 场景 | 视觉策略 |
|------|----------|
| B 端 / 内部工具 | 信息密度高，导航稳定，表格/筛选/批量操作清楚，装饰克制 |
| 教育 / 教师工具 | 任务路径清楚，状态温和但不幼稚，数据和班级/题目/学生信息真实 |
| Agent / Copilot | 对话流 + 状态卡 + 结果预确认并存，用户能看懂 AI 做了什么 |
| C 端 / 移动端 | 轻引导，首屏路径明确，控件触达友好，情绪设计服务任务 |
| 数据分析 / BI | 指标层级、图表解释、筛选上下文、异常状态必须清楚 |
| 营销 / 品牌展示 | 可更强视觉记忆点，但不能牺牲主 CTA 和转化路径 |

## 5. Agent / Hybrid 原型专项

Agent 产品原型必须呈现 6 件事：

1. 用户输入入口：自然语言输入、快捷建议或指令入口。
2. AI 回复：不是纯文本说明，要有结构化结果或行动建议。
3. AI 状态：思考中、检索中、生成中、工具调用中至少一种。
4. 结果预确认：AI 做出决策前，用户能看到摘要并确认/修改。
5. 修改入口：用户能调整 AI 默认值、范围、筛选条件或下一步动作。
6. 失败兜底：数据不足、找不到结果、AI 不确定时给替代方案。

缺任一项，Agent 原型最多只能判"部分达标"。

## 6. 审计评分

每份原型按 12 分制审计：

| 维度 | 0 分 | 2 分 | 4 分 |
|------|------|------|------|
| PRD 覆盖 | 主流程缺失 | 主流程有但状态/子流程缺 | 页面、功能、关键状态覆盖完整 |
| 交互体验 | 静态展示或点击无反馈 | 可点击但流程断裂/反馈弱 | 核心任务可走通，状态反馈明确 |
| 视觉设计 | 模板感强/层级混乱 | 基本可看但缺少场景真实感 | 视觉方向明确，密度、组件、假数据可信 |

判定：
- 总分 >= 10 且单项 >= 3：可进评审。
- 总分 9 或单项 = 2：修完关键问题再评审。
- 总分 < 9 或任一单项 < 2：必须重做或大修。

## 7. 修复 punch list 写法

每条问题必须可执行：

✅ `首页首屏只有欢迎语，缺少教师当天任务入口 → 增加"待处理任务/最近考试/常用动作"三块，并给真实数量。`

❌ `视觉再高级一点。`

✅ `AI 结果只有一段文字 → 改成"摘要卡 + 依据列表 + 确认/修改按钮"，体现预确认。`

❌ `加强 AI 感。`

## 8. 不可越界

- 不把视觉像素、色号、动画毫秒写回 PRD 功能表格。
- 不要求最终生产代码完全复用原型 HTML。
- 不因为是原型就放弃可访问性、焦点态、错误态。
- 未经用户允许，不主动跑 Playwright 相关命令。
