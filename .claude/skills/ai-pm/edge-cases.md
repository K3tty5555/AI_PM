# 边缘情况处理与需求评审

## 边缘情况处理

| 场景 | 处理 |
|------|------|
| 用户只输入一个词（如"记账"） | 深度访谈，挖掘完整需求 |
| 用户说"随便做做" | 追问具体场景和期望 |
| 用户中途想修改需求 | 暂停当前阶段，重新澄清 |
| 某个阶段生成失败 | 提示用户，提供重试选项 |
| 用户想跳过某个阶段 | 询问原因，同意后跳过 |
| 项目名重复 | 提示已存在，询问是否覆盖或重命名 |
| 输出文件夹不存在 | 自动创建 |

## Phase 8: 需求评审详细说明

### 评审流程概述

需求评审是 PRD 和原型完成后的**可选但推荐**的阶段，通过模拟真实的评审会议，让九大专业角色从不同维度审视方案，提前发现问题并优化。

### 触发条件

```
自动触发：原型生成完成后询问用户
手动触发：/ai-pm review
指定轮次：/ai-pm review --round=2
```

### 踩坑提醒（Phase 8 前置检查）

评审开始前，调用知识库检索相关经验（含踩坑记录）：

1. 提取 PRD 功能关键词（产品类型、核心功能点）
2. 调用 `knowledge suggest {关键词}`（统一搜索，覆盖所有知识分类含踩坑）
3. 有匹配 → 展示知识库返回结果（最多3条），用户选看或跳过
4. 无匹配 → 静默

### 评审交互示例

**Phase 7 完成后询问：**
```
原型完成。
  PRD: 05-prd/05-PRD-v1.0.md
  原型: 06-prototype/

要做需求评审吗？（九角色挑刺，提前发现问题）
  开始评审 / 精简评审（4角色）/ 安全专项 / 跳过
```

**Phase 8 执行过程：**
```
阶段 8/8：需求评审（第1轮）
输入：PRD v1.0 + 原型  |  预计：5-8 分钟

[1/8] 研发总监... 发现: 1C 2M 3m
[2/8] 架构师...   发现: 0C 3M 2m
[3/8] 产品总监... 发现: 2C 1M 2m
[4/8] 设计总监... 发现: 0C 2M 4m
[5/8] 项目经理... 发现: 0C 1M 2m
[6/8] QA总监...   发现: 1C 2M 3m
[7/8] 安全合规... 发现: 1C 0M 1m
[8/8] 运营增长... 发现: 0C 2M 2m

第1轮评审完成  5C + 13M + 19m = 37 个问题

Critical（必须修改）：
  1. [QA] 核心功能缺少验收标准
  2. [安全] 个人敏感信息未脱敏
  3. [产品] 用户分层不够细致
  4. [研发] 实时同步技术方案不明确
  5. [QA] 边界情况覆盖不完整

Major（建议修改）：
  1. [架构] 缺少缓存策略设计
  2. [产品] P0 功能过多，建议拆分版本
  3. [项目] 里程碑设置过于密集
  4. [运营] 缺少冷启动内容填充方案
  ...

评审报告 → 08-review-report-v1.md

修改策略：全部修改 / 核心修改（C+M，推荐）/ 最小修改（仅C）/ 查看详情
```

**修改执行过程：**
```
基于评审意见修改（核心修改：C+M）...

修改 PRD...
  + 实时同步技术方案（研发）
  + 用户角色权限细化（产品）
  + P0 功能拆分为 v1.0/v1.1（产品）
  + 缓存策略（架构）  + 验收标准（QA）
  + 敏感信息脱敏（安全）  + 冷启动方案（运营）

更新原型...
  + 空状态设计  + 交互流程优化  + 运营后台入口

更新版本...
  修订日志 +v1.1，直接修改 05-prd/05-PRD-v1.0.md

修改完成。

已解决：Critical 5/5  |  Major 10/13  |  Minor 跳过

主要变更：技术方案 / 安全合规 / 验收标准 / 冷启动运营
          用户角色设计 / 功能优先级 / 里程碑 / 缓存策略

再次评审 / 确认完成 / 查看修改对比
```

### 多轮评审机制

**第一轮评审**：全面审视，发现主要问题
**第二轮评审**：验证修改效果，关注新问题
**终轮评审**：确认达到开发准入标准

**通过标准**：
- Critical 问题全部解决
- Major 问题解决率 ≥ 80%
- 无新引入的 Critical 问题

### 与主流程的集成

```
完整流程（含评审）：
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6 → Phase 7 → Phase 8
参考收集 → 需求澄清 → 需求分析 → 竞品研究 → 用户故事 → PRD生成 → 数据设计 → 原型生成 → 需求评审
                                                                                        ↓
                                                                                  [多轮迭代]
                                                                                        ↓
                                                                                  进入开发

精简流程（跳过评审）：
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 7 → 进入开发
```
