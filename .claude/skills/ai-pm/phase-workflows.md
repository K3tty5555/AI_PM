# 阶段执行流程（优化版）

## 核心原则

- 每个阶段**只确认一次**（开始前或结束后）
- **关键节点**（PRD生成前）必须确认
- **非关键阶段**自动执行，减少打断
- 提供**一键连续模式**（Yolo模式）供熟悉用户使用

## 完整流程图

```
【数据驱动入口 - 可选】
用户上传数据文件 或执行 /ai-pm data {文件}
    ↓
数据洞察阶段（替代传统Phase 0-1）
    ↓ 解析数据文件（Excel/CSV/JSON/Parquet）
    ↓ 探索性数据分析（EDA）
    ↓ 多轮交互提问，挖掘潜在需求
    ↓ 识别用户行为模式、转化问题、流失风险
    ↓ 生成 10-data-insight-report.md
    ↓
数据驱动需求提炼
    ↓ 将数据洞察转化为产品需求
    ↓ 基于问题诊断确定优化方向
    ↓ 生成 11-data-driven-requirements.md
    ↓ 可选择进入标准Phase 2-8流程

Phase 0: 项目启动与参考资源收集（v2.0优化）

步骤1: 项目类型确认
    ↓ 询问用户项目类型：
         - 0-1全新开发
         - 基于已有项目迭代
         - 参考竞品/对标

步骤2: 参考资源收集（关键优化）
    ↓ 主动询问用户是否有参考资源：
         💬 你是否有以下参考资源可以提供？
         📱 现有系统（如果是迭代项目）：现网URL、账号密码、截图
         🎯 竞品参考：竞品网址、截图、关注的功能点
         🎨 设计参考：UI设计稿、品牌规范、参考网站
    ↓ 如果用户提供了资源：
         - URL → 使用playwright-cli自动抓取
         - 截图/图片 → 保存到 07-references/images/
         - 文件 → 保存到 07-references/
    ↓ 生成 07-references/reference-config.md 记录配置
    ↓ 生成 00-reference-analysis.md 分析报告

步骤3: 分析结论确认
    ↓ 基于参考分析，询问："请选择分析模式："
         - 迭代优化 → 侧重现有功能改进
         - 对标开发 → 侧重差异化分析
         - 仅作参考 → 作为竞品研究素材

Phase 1: 需求澄清（三种入口方式）

    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    🎯 入口分支：用户选择如何开始
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    方式A：直接描述需求（标准流程）
        ↓ 检查是否存在 01-requirement-draft.md
             - 存在 → 询问"是否基于现有需求继续？"
             - 不存在 → 开始交互式访谈
        ↓ 实时澄清需求
        ↓ 生成 01-requirement-draft.md

    方式B：用户访谈模式 ⭐面对面场景
        ↓ 调用 ai-pm-interview 技能
        ↓ 结构化访谈提纲
        ↓ 现场记录客户反馈
        ↓ 实时生成 PRD/原型
        ↓ 现场迭代调优
        特点：快、面对面、现场交付

    方式C：数据洞察驱动
        ↓ 调用 ai-pm-data 技能
        ↓ 分析用户行为数据
        ↓ 发现痛点和机会点
        ↓ 基于数据提炼需求
        特点：数据驱动、客观、可量化

    【三种方式都导向：生成 01-requirement-draft.md】
    【自动进入下一阶段，不再单独确认】

Phase 2: 需求分析
    ↓ 静默执行，生成 02-analysis-report.md
    【非关键阶段，自动执行，不询问】

Phase 3: 竞品研究
    ↓ 静默执行，生成 03-competitor-report.md
    【非关键阶段，自动执行，不询问】

Phase 4: 用户故事
    ↓ 静默执行，生成 04-user-stories.md
    【非关键阶段，自动执行，不询问】

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎯 关键确认节点
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Phase 5: PRD 生成前统一确认
    ↓ 汇总展示前4阶段核心结论：
         ```
         确认一下再生成 PRD：
           目标用户：{一句话概括}
           核心痛点：{Top 3 痛点}
           主要功能：{关键功能清单}
           成功指标：{量化指标}

         写作风格：
         【情况A：检测到多种风格】
           1. default（默认）
           2. enterprise-standard（企业标准）
           3. [用户自定义风格]
         【情况B：仅默认风格】
           使用默认风格（/ai-pm config analyze {PRD文件} 可创建自定义风格）

         设计规范：
         【情况A：检测到设计规范】
           1. enterprise-standard
           2. apple-clean
           3. my-company
         【情况B：未检测到】
           使用默认风格（/ai-pm config 可上传规范）

         回复数字选风格/规范，或直接说"生成"。
         要改内容就说"修改XX"，跳过竞品/故事就说"精简"。
         ```

### 知识推荐（Phase 5 前置检查）

生成 PRD 前，检查知识库是否有相关经验：

1. 提取需求关键词（产品类型、核心功能、目标用户）
2. 调用 `knowledge suggest {关键词}`
3. 有匹配 → 展示推荐，用户选查看或跳过
4. 无匹配 → 静默，直接生成 PRD

    ↓ 生成 05-prd/05-PRD-v1.0.md（应用选定的风格+设计规范）

Phase 6: 数据分析与埋点设计【新增】
    ↓ 读取 PRD (05-prd/05-PRD-v1.0.md)
    ↓ 调用 ai-pm-data 技能
         - 指标体系设计（北极星指标+驱动指标）
         - 埋点方案设计（事件定义+参数设计）
         - 数据看板规划（可视化需求）
         - A/B测试方案（实验设计）
    ↓ 询问用户
         ```
         要做数据埋点设计吗？
           内容：指标体系 + 埋点方案 + 看板 + A/B 测试框架

           开始 / 精简版（只做核心指标）/ 跳过
         ```
    ↓ 生成 09-analytics-requirement.md
         - 指标体系
         - 埋点方案
         - 数据看板需求
         - A/B测试方案

Phase 7: 原型生成（Token消耗提示）
    ↓ Token消耗说明
         ```
         下一步：生成可交互原型？
           HTML + CSS + JS，浏览器直接开。
           Token 比 PRD 多，大概 1-3 分钟。

           生成原型 / 跳过（先评审 PRD 的话建议跳过）
         ```
    ↓ 询问用户：是否生成原型？
         - 生成原型 → 进入Phase 6执行
         - 跳过 → 输出完成总结

Phase 8: 原型生成（执行）
    ↓ 读取 PRD 和数据分析文档
         - 从 05-prd/05-PRD-v1.0.md 读取设计规范
         - 从 09-analytics-requirement.md 读取埋点需求
         - 加载 templates/ui-specs/{规范名}/design-tokens.json
    ↓ 询问设备类型（如未指定）
         ```
         设备类型？Web 端（默认）/ 移动端 / 响应式
         ```
    ↓ 生成 06-prototype/（可交互网页原型）
         - 应用 PRD 阶段选定的设计规范
         - 集成数据分析阶段的埋点需求
         - 应用指定的设备类型
    ↓ 输出完成总结
    ↓ 询问是否进入需求评审
         ```
         原型生成完了。要做需求评审吗？（九个角色轮番挑刺）
           开始评审 / 跳过
         ```

Phase 8: 需求评审（可选，多轮迭代）
    ↓ 读取 PRD (05-prd/05-PRD-v1.0.md) 和原型 (06-prototype/)
    ↓ 调用 ai-pm-review 技能
         - 研发总监评审（技术可行性、开发成本）
         - 资深架构师评审（架构设计、扩展性）
         - 产品总监评审（产品定位、商业价值）
         - 设计总监评审（用户体验、交互设计）
         - 项目经理评审（项目排期、风险管理）
         - QA总监评审（可测试性、验收标准）
         - 安全合规官评审（数据安全、合规要求）
         - 运营增长官评审（冷启动、数据埋点）
         - 数据分析师评审（指标体系、埋点方案）
    ↓ 生成 08-review-report-v{N}.md
    ↓ 用户确认修改范围
         - 全部修改 / 核心修改 / 最小修改 / 指定修改
    ↓ 基于评审意见修改 PRD（v2.0优化：直接在原文档更新）
         - 在修订日志中增加新版本记录
         - 直接在 05-prd/05-PRD-v1.0.md 中修改内容
         - 不创建新文件，保持文档唯一性
         - 更新 06-prototype/（如有需要）
    ↓ 生成修改说明文档 08-review-changes.md（可选）
    ↓ 询问是否再次评审
         - 再次评审 → 生成 08-review-report-v{N+1}.md
         - 确认完成 → 流程结束，进入开发阶段
```

### 知识沉淀（项目完成后）

任何完成路径（含 Phase 7 跳过评审、Phase 8 评审结束）均触发：

1. 执行 `knowledge sync`，扫描项目产出
2. 展示候选知识点，用户选择入库
3. 确认后批量写入 `templates/knowledge-base/`
4. 跳过时提示：「随时可用 `/ai-pm knowledge sync` 补充」

## 用户体验优化点

| 优化项 | 原流程 | 新流程 | 效果 |
|--------|--------|--------|------|
| Phase 2-4确认 | 每个阶段前后都问（6次确认） | 静默执行，不询问 | 减少6次打断 |
| Phase 1确认 | 生成后问一次 | 存在草稿时问一次，否则直接开始 | 更自然 |
| 关键节点 | 分散在各阶段 | 集中在PRD前统一确认 | 用户掌控感强 |
| 原型生成 | 可能有重复确认 | Token提示+一次确认 | 成本透明 |

## 各阶段快捷操作

在每个阶段确认节点，提供统一的快捷操作提示：

```
  继续 / go → 下一步
  跳过 → 跳过当前阶段
  看PRD / 看原型 / 看分析 / 看竞品 → 查看对应内容
  状态 → 项目仪表盘
  列表 → 所有项目
```

## Yolo模式（快速通道）

对于熟悉流程的用户，可在开始时选择：
```
执行模式：
  标准 → 关键节点确认，第一次用推荐这个
  Yolo → 自动跑到 PRD 前，熟悉流程可以用
```

Yolo模式下：
- Phase 1-4 全自动执行
- 仅在PRD生成前停一次
- 询问原型生成时停一次
