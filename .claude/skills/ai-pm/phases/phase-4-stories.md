# Phase 4: 用户故事

**输入**: `02-analysis-report.md` + `03-competitor-report.md`
**输出**: `04-user-stories.md`

## 执行方式

静默执行。基于需求分析和竞品研究，自动拆解用户故事并编写验收标准。

用户故事格式：

```
作为 {用户角色}，
我希望 {功能描述}，
以便 {价值/目的}。

验收标准：
- [ ] {可量化的验收条件}
- [ ] ...
```

---

## INVEST 质量门控（落盘前执行）

`04-user-stories.md` 写入后，立即对每条故事做 INVEST 自检：

| 维度 | 检查问题 | 不达标标准 |
|------|---------|-----------|
| **I**ndependent | 此故事可以独立开发吗？ | 强依赖另一条未完成故事 |
| **N**egotiable | 这是约束还是解决方案？ | 已固化了技术实现方式 |
| **V**aluable | 对用户有明确价值吗？ | 「以便」部分是空话或无法量化 |
| **E**stimable | 能估算工作量吗？ | 描述过于模糊，无法给出工作日估算 |
| **S**mall | 一个 sprint 内能完成吗？ | 验收标准超过 5 条 |
| **T**estable | 验收标准可以被测试验收吗？ | 包含「系统应该」「用户应该」等模糊表述 |

### 检查后展示结果

```
📋 用户故事质量检查（INVEST）

✅ US-01 作为教师，我希望查看考试成绩分布...
   I✅ N✅ V✅ E✅ S✅ T✅

⚠️ US-03 作为管理员，我希望系统有良好的性能...
   I✅ N✅ V⚠️（「良好」无法量化） E⚠️ S✅ T⚠️（无可测试验收标准）

❌ US-07 作为用户，我希望使用微服务架构...
   N❌（固化了技术实现）

通过：{N} 条，待改进：{M} 条，不达标：{K} 条
```

### 用户确认分支

- **全部通过（M+K=0）** → 静默更新 `_status.json`（phases.stories = true），输出「✅ 用户故事质量检查通过，共 N 条故事」
- **有待改进/不达标（M+K>0）** → 展示上述结果后询问：

```
⚠️ 有 {M+K} 条故事需要确认，如何处理？
  1. 继续（带瑕疵进入 PRD）
  2. 修改（告诉我哪条需要怎么改）
```

用户选「继续」→ 更新状态，进入 PRD 流程
用户选「修改」→ 引导修改，修改完成后**重新执行 INVEST 检查**

## cost 写入

stories 阶段完成后，估算 token 消耗并写入 `_status.json`：

```bash
wc -c {project_dir}/04-user-stories.md
```

```json
"cost": {
  "phases": {
    "stories": {
      "model": "claude-sonnet-4-6",
      "tokens_estimate": {file_bytes * 0.25},
      "completed_at": "{ISO8601}"
    }
  }
}
```

并更新 `cost.total_estimate`（累加所有已有 phases 的 tokens_estimate）。
