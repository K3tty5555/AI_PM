# 竞品研究

## 输入

- 主要：`{项目目录}/02-analysis-report.md`（需求分析报告）
- 或：用户直接描述的产品方向

## 输出

`{项目目录}/03-competitor-report.md`（竞品分析报告）

## 执行步骤

### 步骤1：读取需求分析，识别竞品方向

从需求分析中提取：产品类型、目标用户、核心功能方向，据此识别竞品范围。

**竞品分类：**
- 直接竞品：解决相同问题、面向相同用户
- 间接竞品：解决相同问题、方式不同
- 参考竞品：不同领域但有可借鉴之处

### 步骤2：询问用户补充

```
基于需求分析，我计划研究以下竞品：
{列出识别的竞品清单}

请问：
1. 有没有你特别关注的竞品需要深入分析？
2. 哪个竞品做得好？哪个有明显缺陷？
3. 有没有跨领域的优秀产品值得参考？

请直接告诉我，或回复"按你的分析来"。
```

### 步骤3：研究每个竞品

**信息来源：** 官网、应用商店（评分/评论）、行业报告、用户社区反馈

**研究维度：**
- 功能：核心功能、特色功能、功能完整度
- 体验：交互设计、易用性、视觉风格
- 商业：定价策略、盈利模式、用户规模
- 技术：技术路线、性能表现、创新点

#### Playwright MCP 调用策略

竞品网页分析涉及多个竞品官网的内容抓取，必须遵循以下策略以提升效率和稳定性。

**批量抓取模式（同会话复用）：**

竞品网页分析时，在同一个 Playwright 会话中完成所有页面的抓取：
1. 先用 `browser_navigate` 打开第一个竞品页面
2. 用 `browser_snapshot` 获取内容
3. 直接 `browser_navigate` 到下一个竞品页面（不关闭浏览器）
4. 重复直到所有页面抓取完毕

这样比每个页面单独启动 Playwright 会话快 3-5 倍。

**重试与超时：**

- 单页面抓取超时设置：30 秒
- 超时后自动重试一次（共 2 次尝试）
- 仍失败则降级为 `WebFetch`（纯 HTTP GET + HTML 标签剥离）并继续
- 不因单个页面失败而中断整个竞品分析

**降级策略（Playwright MCP 不可用时）：**

如果 Playwright MCP 不可用（未安装或连接失败）：
1. 自动降级为 `WebFetch` 命令（纯 HTTP GET + HTML 标签剥离）
2. 向用户明确提示："已降级为纯文本抓取。JS 渲染内容（如 SPA 页面）可能无法正确获取，分析结果可能不完整。"
3. 继续执行分析流程，基于纯文本内容完成竞品研究

### 步骤4：制作对比矩阵，识别差异化机会

汇总所有竞品对比，识别：
- 市场空白（未被满足的需求）
- 竞品弱点（可以突破的点）
- 我方优势（可以发挥的长处）
